# moka-1 Design Language

> **コンセプト: 瑠璃と金泥 — 紺紙金泥の様式に倣う**
> 紺の紙に金泥で経を写す「紺紙金泥」。青は読みの静けさ、金は理解の光。
> moka-1 の画面は瑠璃系の濃淡だけで組まれた青の紙であり、
> AI(moka)の印だけが金泥で記される。
> AI はカウンターの向こうにいる給仕であり、主役は常に記事と読者である。

- **Version**: 3.3.0 (2026-07-18) — 対訳(§5)と訊く(§8.2)を実装再導入まで取り下げ(規定自体は保持)。記事リスト行の手がかり(フィード名・相対日時・既読の沈み)を規定(§8.6)。フィード削除に警告様式の二段確認を適用(§8.7)。ホーム空状態にハイライトの静的予告文を追加(§7.2)。要約カードの既定状態とストリーミング中の面の安定を明文化(§8.1)。前版: 3.2.0 — §4.3 モバイルUXをドロワーから一覧↔読書のマスター・ディテールプッシュ遷移に変更。前々版: 3.1.0 — ドリップを待ち全般(推論・取り寄せ)に拡張、間のリズム・墨流し・グーイー合流に刷新(§6.2)、金の一滴(推論完了専用)を追加(§6.3)。3.0.0 — 色相を瑠璃・群青系へ転換、第二色相として金泥のみを許可
- **Scope**: moka-1 Web UI 全体(デスクトップ / モバイル)
- **Theme**: ライト専用。ダークモードは提供しない(トークンにダーク値を持たない)

## 1. デザイン原則

迷ったときはここに立ち返る。優先順位は番号順。

1. **記事が主役、AI は給仕**
   要約・翻訳・Q&A は「運ばれてくる」ものであり、読書の流れを遮らない。AI 要素が本文より視覚的に強くなったら、それは設計ミスである。

2. **書体と濃淡が語り手を示す**
   明朝 = 原文・記事の声。ゴシック = moka と UI の声。AI 生成物は常に `fujinezu` の面に載り、moka の印(ラベル)は `kindei` で記される。引用符・ロボットアイコン・「AI生成」バッジのような冗長な標識に頼らない。

3. **青一色、金は印にのみ**
   moka-1 は瑠璃系一色相の濃淡で構成され、第二色相は **金泥ただ一つ**。金は moka の印・特別な強調・紺紙金泥ブロック(§2.4)に限り、面積を持つ装飾には使わない。赤・黄・緑・茶は存在しない。純白・純黒も使わない。

4. **待ち時間は蒸らし時間**
   ローカル LLM の推論待ちを回転スピナーで急かさない。ドリップの滴りと工程の言葉(§7)で「じっくり抽出している」体験に変える。待ち時間は弱点ではなく世界観の一部。

5. **瑠璃は一滴、金は一筋**
   インタラクション色 `ruri` は 1 画面(1 ビューポート)につき原則 1 箇所。`kindei` は moka の印としてのみ現れ、1 画面に複数の金がひしめいてはならない。

## 2. カラー

### 2.1 瑠璃の濃淡スケール(淡 → 濃)

名は青の和色から(画面表示用に明度・彩度は調整済み)。

| Token       | 和名 | Hex       | 役割                                                         |
| ----------- | ---- | --------- | ------------------------------------------------------------ |
| `geppaku`   | 月白 | `#F4F6FB` | 浮いた面(カード、ヘッダー、入力欄)。最も明るいトーン         |
| `kumoi`     | 雲居 | `#EDF0F7` | ページ地。アプリの背景はすべてこの色から始まる               |
| `fujinezu`  | 藤鼠 | `#DFE4F1` | **AI の声の面**。要約・翻訳・Q&A 回答は必ずこの面に載る      |
| `hatoba`    | 鳩羽 | `#C9CFE2` | 罫線・区切り(0.5〜1px hairline)、選択面                      |
| `usuhanada` | 薄縹 | `#94A0BE` | プレースホルダ・無効状態。**本質的情報には使用禁止**         |
| `konnezu`   | 紺鼠 | `#545C78` | 補助テキスト(メタ情報、キャプション)                         |
| `ruri`      | 瑠璃 | `#3E4D8E` | インタラクション(リンク、主ボタン、フォーカス、ドリップの滴) |
| `kon`       | 紺   | `#1F2438` | 本文テキスト。最も濃いトーン。紺紙金泥ブロックの地           |

補助トーン: `ruri-deep` `#313E77`(ruri の hover / active)、`ruri-tint` `#D6DBF0`(押下面・選択強調)。

### 2.2 金泥(第二色相 — 唯一の例外)

| Token           | Hex       | 役割                                                             |
| --------------- | --------- | ---------------------------------------------------------------- |
| `kindei`        | `#745A17` | 文字用の金。moka の印(「moka による要約」等のラベル)、警告枠     |
| `kindei-bright` | `#C9A227` | 装飾用の金。**kon 地の上でのみ使用**(紺紙金泥ブロックの文字・線) |

金泥の規律: 淡い地の上では `kindei`(暗い金)のみ、`kindei-bright` は紺の地の上のみ。金のグラデーション・メタリック表現は禁止(フラットな一色として扱う)。

### 2.3 コントラスト実測値

| 組み合わせ              | 比       | 判定                                      |
| ----------------------- | -------- | ----------------------------------------- |
| `kon` / `kumoi`         | 約 13:1  | AAA(本文に使用可)                         |
| `konnezu` / `kumoi`     | 約 5.8:1 | AA(補助テキストに使用可)                  |
| `ruri` / `kumoi`        | 約 6.9:1 | AA+(通常テキスト可)                       |
| `kindei` / `kumoi`      | 約 5.7:1 | AA(ラベル可)                              |
| `kindei` / `fujinezu`   | 約 5.1:1 | AA(AI 面上のラベル可)                     |
| `kindei-bright` / `kon` | 約 6.4:1 | AA+(紺紙金泥ブロック)                     |
| `usuhanada` / `kumoi`   | 約 2.3:1 | 不合格 — プレースホルダ等の非本質要素限定 |

### 2.4 状態表現

状態は **濃淡 + アイコン + 文言** の 3 チャンネルで伝える(色だけに頼らない: WCAG 1.4.1)。

| 状態       | 表現                                                                                                                             |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 成功・完了 | `fujinezu` 面 + チェックアイコン + 完了文言(例:「保存しました」)                                                                 |
| 警告       | `geppaku` 面 + `kindei` 1px 枠 + 警告アイコン + 「注意:」接頭辞                                                                  |
| エラー     | **紺紙金泥ブロック**: `kon` 地 + `kindei-bright` 文字 + アイコン + 「失敗:」接頭辞。画面内で最も荘厳な表現が自動的にエラーへ向く |
| 進行中     | ドリップインジケータ + 工程コピー(§7)                                                                                            |

### 2.5 禁止事項

- 金泥以外の第二色相の導入(赤・黄・緑・茶・青緑を含む一切)
- 淡地の上での `kindei-bright`、面積を持つ金の面、金のグラデーション/メタリック表現
- 純白(#FFFFFF)・純黒(#000000)の使用
- グラデーション、ドロップシャドウ(フォーカスリングを除く)、ぼかし、グロー
- 暖色クリーム地への回帰。「クリーム地+茶文字+くすみ緑/クレイのアクセント」は AI 生成 UI と 2026 年トレンドの二重の定番であり、moka-1 の個性はそこからの距離で成立している(§付記)

## 3. タイポグラフィ

### 3.1 書体ファミリー

| 役割                   | 書体                                       | 備考                                   |
| ---------------------- | ------------------------------------------ | -------------------------------------- |
| 記事の声(本文・見出し) | **Shippori Mincho** (400 / 500)            | 原文・訳文・記事タイトル               |
| moka と UI の声        | **Zen Kaku Gothic New** (400 / 500)        | ナビ、ボタン、要約、Q&A 回答、メタ情報 |
| コード                 | **IBM Plex Mono** (400 / 500 / 400 italic) | 記事内コードブロック、インラインコード |

ウェイトは全書体 **400 と 500 の 2 種のみ**。600 以上は使わない。

フォントは **セルフホスト**(woff2 を moka-1 自身が配信)。AIX 上の自己ホスト型アプリとして、外部 CDN 依存とオフライン時の書体崩れを避ける。

### 3.2 タイプスケール

| 用途               | サイズ   | 行間     | 書体           |
| ------------------ | -------- | -------- | -------------- |
| 記事タイトル       | 22px     | 1.6      | Mincho 500     |
| 見出し(記事内 h2)  | 18px     | 1.7      | Mincho 500     |
| **記事本文**       | **15px** | **2.05** | Mincho 400     |
| UI 本文・moka の声 | 14px     | 1.8      | Gothic 400     |
| メタ情報・ラベル   | 12px     | 1.6      | Gothic 400/500 |
| 最小(バッジ等)     | 11px     | 1.5      | Gothic 500     |
| コード             | 13px     | 1.7      | Plex Mono 400  |

11px 未満は禁止。

### 3.3 読書測度

- 和文本文の最大行長は **約 38 字**(`max-width: 38rem` 相当を目安に調整)
- モバイルでは全幅(左右パディング 20px)
- 段落間は行送り 1 行分(`margin-block: 1em`)。字下げはしない

## 4. レイアウトと余白

### 4.1 スペーシング

4px 基数のスケール: `4 / 8 / 12 / 16 / 24 / 32 / 48 / 64`。
垂直リズムは rem(1rem = 16px)、コンポーネント内部は px で指定する。

### 4.2 角丸

| 対象                                | 値               |
| ----------------------------------- | ---------------- |
| ボタン・入力欄・チップ              | 8px              |
| カード(要約、Q&A、紺紙金泥ブロック) | 12px             |
| 入力バー(訊く…)                     | pill(高さの 1/2) |

片側ボーダー(border-left 等)によるアクセントは角丸と併用しない。

### 4.3 画面構成(中庸密度)

```
デスクトップ (>= 900px)
┌────────┬──────────────────────┐
│ サイドバー │  読書カラム(最大 38 字)      │
│ 280px   │  ヘッダー / 本文 / 対訳       │
│ 記事リスト │                        │
│ 履歴     │  [この記事について訊く…]      │
└────────┴──────────────────────┘

モバイル (< 900px) — 記事未選択(記事リスト、フル画面)
┌──────────────────────┐
│ moka-1              … │ ← topbar(状態で中身が変わる)
│ 記事リスト(全幅)              │
│ …                          │
└──────────────────────┘

モバイル (< 900px) — 記事選択後(読書ビュー、フル画面)
┌──────────────────────┐
│ ← 戻る                  │ ← topbar(同じ枠、中身は戻るリンクに)
│ 読書カラム(全幅)              │
│ …                          │
│ [この記事について訊く…] ← 下部固定 │
└──────────────────────┘
```

- ブレークポイントは **900px の 1 つだけ**。中間状態を作らない
- モバイルはドロワーを持たない。記事リストと読書ビューはマスター・ディテール式のプッシュ遷移(スライド、`dur-calm`/`ease-calm`)で切り替わる一枚絵。記事リストは記事未選択時にフル画面表示、記事選択で読書ビューに置き換わる
- 「戻る」は常に記事リストへの明示リンク。記事リストのスクロール位置・無限スクロールの追加読み込み分は遷移をまたいで保持される
- topbar の「…」はポップオーバーで「フィード管理」への導線を収める
- Q&A 入力バーはモバイルで画面下部に固定(safe-area-inset 考慮)— ただし訊くは取り下げ中(§8.2)で、現行の読書ビューに下部固定ドックは存在しない

## 5. 翻訳ビュー(対訳)

> **取り下げ中(v3.3.0)**: LLM 翻訳の実装が入るまで、対訳の切替は画面から取り下げている。
> 「まだ訳されていません」だけを運ぶ給仕は置かない(準備ができていない皿は出さない)。
> 本節の規定は再導入時のためにそのまま保持する。

moka-1 の中核機能。**左右対訳を正とし、モバイルは段落交互に降格する。**

### 5.1 デスクトップ: 左右対訳

- 2 カラムグリッド(`1fr / 1fr`、gap 32px)。左 = 原文、右 = 訳文
- **段落単位で高さを同期**する(段落ペアを 1 行のグリッドアイテムとして揃える)。文単位の同期はしない
- 原文・訳文とも Shippori Mincho(どちらも「記事の声」であるため)
- 訳文は AI 生成物なので、訳文カラム全体を `fujinezu` の面に載せる。これにより原則 2 を維持する
- ホバー中の段落ペアを `hatoba` で淡く同期ハイライト

### 5.2 モバイル: 段落交互

- 原文段落(kumoi 地)→ 訳文段落(fujinezu 面、12px 角丸)の順で交互に積む
- 訳文ブロックの先頭に `kindei` の小ラベル「訳」(11px Gothic 500)— 金泥の印

### 5.3 共通

- 未訳段落には fujinezu 面のスケルトンではなく、**ドリップインジケータ(§6)を段落位置に置く**
- 原文↔訳文の表示切替はヘッダーの言語アイコンから(全文一括)

## 6. モーション

### 6.1 基本則

| Token      | 値                                           | 用途                       |
| ---------- | -------------------------------------------- | -------------------------- |
| `dur-fast` | 120ms                                        | hover、フォーカス          |
| `dur-base` | 200ms                                        | 面の出入り、アコーディオン |
| `dur-calm` | 300ms                                        | ドロワー、対訳切替         |
| easing     | `ease-out`(`cubic-bezier(0.22, 1, 0.36, 1)`) | 全域                       |

- **回転スピナーは全面禁止**
- `prefers-reduced-motion: reduce` 時はドリップを含む全アニメーションを停止し、テキストのみで状態を伝える

### 6.2 署名アニメーション: ドリップ

moka-1 で唯一の装飾的アニメーション。待ち全般(推論・取り寄せ)の間、`ruri` の滴が不等間隔に落ち、
着水のたびに墨が非対称に拡がって消える。金泥はループでは使わない(金は動かない) —
金の表情は推論完了の瞬間だけの静止表現(§6.3)。

**間のリズム**: 3 滴 / 5.4 秒周期を均等割りしない。長い沈黙(約 1.9 秒)の後に 2 滴が短い間隔
(約 0.4〜0.7 秒)で続く — ネルドリップの最後の絞りに似た不規則さ。既存のコーヒー系ローダーの
多くが等間隔ループであるのに対し、沈黙そのものを設計素材にする点で異なる。

**着水のグーイー合流**: 滴が液面に触れる瞬間(18%→20% 付近)、SVG フィルタ(`feGaussianBlur` +
`feColorMatrix` のコントラスト強調)で滴と液面を視覚的に融合させてから滴を消す。表面張力で
吸収されるような物理感を、`transform`/`opacity` だけで動かす子要素に静的フィルタをかぶせて作る
(フィルタ自体はアニメーションしない — パフォーマンスのため)。

**墨流しの着水**: 合流の直後、液面の中の 1 個の楕円が非対称な向き(毎回符号を変える)へ
`scale` + `translateX` で拡がって消える。同心円のリップルにはしない — 墨が水に触れて
一方向へ流れる動きを模す。

**液面の呼吸**: 液面は静止時 `opacity: 0.35` に沈め、着水の瞬間(間のリズムと同期)だけ
`opacity: 1` に明るくなる。沈黙の時間が長いぶん、常時全開のままだと「孤立した線」に
見えてしまうため — 液面自体も間のリズムに参加させる。

### 6.3 金の一滴(推論完了専用)

推論(要約・Q&A 等、M1 以降)が完了した瞬間だけ、ドリップの滴と液面が `kindei-bright` の
静止した金へ切り替わる。ループのアニメーションではなく、完了の一度きりの状態 — 金は動かない
という規律をここでも守る。取り寄せ(ネットワーク待ち)には使わない。実装は
`DripIndicator` の `completed` prop。

## 7. moka の声(Voice & Tone)

moka は静かな給仕である。丁寧だが簡潔、絵文字を使わず、感嘆符を使わない。「です・ます」調。

色彩は紺紙金泥(写しの様式)、工程の言葉は珈琲(ドリップ)。moka-1 は「静かに読む場所」としてこの二つの系譜を引き受ける。

### 7.1 推論工程のコピー(llama-swap の状態に対応)

| システム状態                       | 表示コピー       |
| ---------------------------------- | ---------------- |
| モデルロード中                     | 豆を挽いています |
| モデル切替中(llama-swap swap)      | 豆を替えています |
| プロンプト評価中(prefill)          | 蒸らしています   |
| トークン生成中(streaming 開始まで) | 注いでいます     |

工程コピーは常にドリップインジケータと併記する。技術用語(モデル名、トークン数)はこの文言に混ぜない — 詳細は設定画面や開発者向け表示に隔離する。

### 7.2 コピー規則

- ラベル・ボタンは動詞から。「要約する」「訳す」— 「OK」「送信」は使わない
- エラーは事実 + 次の行動。謝罪しない。例: 「失敗: モデルの読み込みに失敗しました。再試行してください」
- 空状態は招待。例: 「まだ記事がありません。URL を貼るとここに並びます」
- 成功トーストに「成功しました」と書かない。「保存しました」で足りる
- ホームの読書カラムの空状態には、招待に添えて**今日のハイライトの静的予告文を一文だけ**置く: 「やがて毎朝、今日読むべき記事がここに揃います」。操作要素・進行アニメーションは付けない(まだ淹れていないものを湯気で誇示しない)

## 8. コンポーネント規定

- ストリーミング(逐次表示)中は fujinezu 面に **min-height を与えて確保**し、トークン到着でレイアウトが跳ねないようにする

### 8.2 Q&A(訊く)

### 8.3 コードブロック(記事内)

- IBM Plex Mono 13px、`geppaku` 地、`hatoba` ボーダー、8px 角丸
- シンタックスハイライト: キーワード = ruri 500、文字列 = kindei、コメント = usuhanada italic、その他 = kon。トーン + ウェイト + イタリックで区別し、これ以外の色を持ち込まない

### 8.4 トースト / エラー

- 画面下部中央、12px 角丸
- 通常トースト: geppaku 地 + kon 文字 + 状態アイコン
- エラー: **紺紙金泥ブロック**(kon 地 + kindei-bright 文字 + アイコン + 「失敗:」)
- 3.5 秒で自動退場(dur-base のフェード)。エラーのみ手動で閉じるまで残る

### 8.5 フォーカスと操作性

- フォーカスリング: `0 0 0 2px kumoi, 0 0 0 4px ruri`(唯一許可されるシャドウ)
- タップターゲットは最小 44x44px(モバイル)
- すべての操作はキーボードで到達可能であること
- 状態を色(濃淡)だけで伝えない。常にアイコンまたは文言を併記する(§2.4)

### 8.6 記事リスト行(サイドバー)

リスト行は UI の声(ゴシック)。手がかりはタイポグラフィの階層だけで作り、枠・面・バッジを足さない。

- タイトル: 14px Gothic、**未読 = kon 500 / 既読 = konnezu 400**(既読の沈み)。打ち消し線・チェックマークは使わない
- メタ行: **フィード名 + 相対日時**(12px Gothic 400 / konnezu)を「・」で結ぶ。フィード名が無ければ記事 URL のホスト名で代替する
- 相対日時: 「たった今」「N分前」「N時間前」「昨日」「N日前」。おおむね1週間を超えたら絶対日付(「2026年7月1日」)に切り替える。感嘆符は使わない
- **未読数・バッジは一切置かない**(急かさない店)。既読は濃淡で静かに沈めるのみ
- 行の区切りは hatoba の hairline のみ。読書中の行は `ruri-tint`、hover は `hatoba`(従来どおり)

### 8.7 フィード削除の確認(警告様式の適用)

フィードの削除は破壊的(その店の記事・要約・既読の事実ごと消える、ADR00019)。

- 削除の入口は一覧行の静かな二次操作(面・枠なしのテキストボタン)。一発では発火しない
- 確認は**警告様式(§2.4)のインラインブロック**: `geppaku` 面 + `kindei` 1px 枠 + 警告アイコン + 「注意:」接頭辞
- 確認文言は記事も消える事実を明示する: 「注意: このフィードを削除すると、届いた記事・要約・既読の記録もすべて消えます。元に戻せません」
- 紺紙金泥ブロックは**エラー専用**のまま — 確認には使わない
- 成功トーストは「削除しました」(§7.2)
