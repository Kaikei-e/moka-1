# 交代枠モデルの第一候補を Qwen3.6-35B-A3B に更新する

## Context

tenets §3.3 の LLM 編成は、品質最優先バッチ(週次リキャップ、RAG 最終回答の高品質版)だけにロードする**交代枠**として Qwen3.5-35B-A3B(~22GB)を挙げていた。これは draft_0001(2026-07-02 の LFM2.5 調査)時点の情報に基づく。

同日の Web 調査で、後継の **Qwen3.6-35B-A3B**(2026-04-15 公開、Apache-2.0)が既に出ていることを確認した:

- **同一フットプリント**: 総 35B / アクティブ 3B の MoE 構成は据え置き。unsloth の UD-Q4_K_XL で 20.82 GiB(3.5 の同量子化 20.71 GiB とほぼ同じ)。4bit 時の VRAM 目安 ~23GB は単独ロードなら iGPU 32GB に収まる
- **能力の段差**: SWE-bench Verified 73.4%(同アクティブ級の従来 open SOTA は 60% 未満)。エージェント的タスクと推論での改善が公称の主軸
- **ネイティブマルチモーダル化**: vision 対応(GGUF は mmproj 別ファイル)。コンテキスト 262K(拡張時 ~1M)
- **エコシステム成熟**: HF ダウンロード数 568 万超、unsloth の Dynamic 量子化・推奨サンプリング(思考モード temp 1.0 / top_p 0.95 / top_k 20、コーディングは temp 0.6)が整備済み
- **注意点**: MTP(Multi-Token Prediction)変種は `-np > 1` と併用不可の報告があった(現在は mainline で解消と報告されているが、moka-1 は `-np` バッチ並列が前提なので要実測)

moka-1 はまだ M0(スケルトン)段階で、交代枠モデルは配線どころか一度も推論していない。「どのモデルを事前ダウンロードしておくか」を決めるだけの局面であり、切替コストが最も低い今が更新のタイミングになる。

## Decision

**交代枠の第一候補を Qwen3.6-35B-A3B(`unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL`、20.82 GiB)に更新する。**ただし:

1. **正式採用は eval/ での実測後**。Qwen3.5-35B-A3B(UD-Q4_K_XL)もキャッシュに保持し、moka-1 の実タスク(日本語要約・リキャップ品質)で比較してから tenets §3.3 を確定させる。llama.cpp pin(b9859)での動作確認もこの実測に含める
   - **(追記 2026-07-02 同日)速度側は実測済み**: b9859 の Vulkan(RADV gfx1150)で両者ともロード・生成を確認。`llama bench -ngl 99 -r 2` で 3.6 = pp512 343.4 / tg128 23.3 t/s、3.5 = pp512 350.2 / tg128 23.4 t/s と**速度は同等**(llama.cpp は両者を同一アーキテクチャ `qwen35moe` として扱う)。乗り換えの速度ペナルティは無し。**残る採用条件は品質 A/B のみ**([[ADR00006]] に全モデルの実測表)
2. **テキスト専用で使う**。mmproj(vision)と MTP 変種は当面ダウンロードも配線もしない。RSS リーダーのコアタスクに vision は不要で、MTP は `-np` との相性が未確認(tenets のミニマリズム: 使わない能力は持ち込まない)
3. **提案層は Qwen3.5-4B のまま**。Qwen3.6 の open weights は 27B dense と 35B-A3B のみで、4B 級の後継が存在しない

**検討した代替案と却下理由**:

- **Qwen3.5-35B-A3B を維持(tenets 通り)**: 同サイズ・同ライセンス・同配布元で世代だけ新しい選択肢があるのに、まだ何も配線していない段階で旧世代に固定する理由がない。「モデル更新は eval とセットで意図的に」の原則は、比較実測を採用条件にすることで満たす
- **Qwen3.6-27B(dense)**: 4bit で ~18GB とやや小さいが、dense ゆえデコードが A3B 系より大幅に遅い(交代枠でもリキャップは長文生成なので速度は効く)。MoE の「メモリは総パラ・速度はアクティブ」の利得(draft_0001 §2.1 と同じ論理)を捨てる理由がない
- **MTP 変種の採用**: 投機的デコードで単発は速くなるが、`-np` バッチ並列との併用実績が薄い。安定性を旗印にする交代枠に実験的構成は不適
- **Qwen3.7-Max 系(ホスト版)**: API 提供のみ。ローカル LLM 限定(tenets §2-5)に反する

## Consequences

### Pros

- 同一ディスク/VRAM 予算のまま、交代枠の期待品質が一世代分上がる(公称ベンチでは大差)
- 262K コンテキストは週次リキャップ(1週間分の記事要約をまとめて食わせる)で余裕が生まれる
- Apache-2.0・unsloth Dynamic 量子化・推奨パラメータ整備済みで、採用時の運用調査コストが低い

### Cons / Tradeoffs

- モデルキャッシュが一時的に +20.8 GiB(3.5 と 3.6 の併存)。eval 決着後に敗者を消す運用が必要
- 公称ベンチはコーディング/エージェント寄りで、moka-1 の主用途(日本語要約・編集)への転移は未実証 — eval/ の実測が済むまで「候補」止まり
- tenets §3.3 の表と本 ADR の間に不整合期間が生まれる(tenets 更新は eval 決着とセットで行う)

## Revisit Triggers

- eval/ の比較実測で Qwen3.5-35B-A3B と品質差が誤差範囲、または Vulkan 実速度・`-np` 併用で 3.6 側に退行があれば、3.5 維持に戻す
- llama.cpp pin 更新時に Qwen3.6 系の既知問題(MTP・テンプレート)が解消/悪化したら構成を再評価
- 4B 級の Qwen3.6/3.7 open weights が出たら提案層の更新も同じ手順(事前 DL → eval → tenets 反映)で検討
- 週次リキャップの品質が高速パス(LFM2.5)+提案層だけで足りると分かったら、交代枠自体の廃止を検討(ミニマリズム)
