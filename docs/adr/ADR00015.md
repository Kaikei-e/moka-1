# 要約プロンプトをA/Bで改善しストリーミング応答を専用SSEエンドポイントで実装する

## Context

ADR00014 で要約機能を実装した後、ユーザーから「要約の質が悪い」という指摘があり、(1) プロンプトチューニングと (2) ストリーミング表示の2点を改善することになった。

調査の過程で2つの既存の問題を発見した。

1. **eval と本番プロンプトの乖離**: ADR00014 の実装中、本番プロンプト(`core/internal/summarize/client.go`)は「3〜5文で簡潔に」から「400字程度(目安8〜12文)」へユーザーフィードバックを受けて手動チューニングされたが、`eval/prompts/summarize_ja.txt`(評価ハーネス用)には反映されず「3〜5文」のまま放置されていた。ADR00014 の Cons にも「400字目標は経験則によるチューニングであり、定量A/Bを経ていない」と明記されていた。
2. **ストリーミング未対応**: `POST /api/v1/articles/{id}/summary` は非ストリーミングで、生成完了まで読者に何も表示されない。

Web調査(2026-07時点)では、小型ローカルLLMの要約タスクにおいて Chain of Density 等の多段プロンプト手法は複数回のLLM呼び出しを要し、ADR00014 §7 の「リトライ・サーキットブレーカー無し・単発呼び出し」というミニマリズム方針と衝突することを確認した。単発呼び出しで効果が見込めるのは「出力構造の明示(背景・具体的事実・結論を列挙要件にする)」と「否定制約の強化(一般論での字数埋め禁止)」という方向性だった。

ストリーミング実装前に `/grill-with-docs` で tenets・ADR00014 との整合性を検討し、ユーザーに以下の設計論点を選択式で確認した(詳細は Decision 参照): ストリーミングAPIの設計(同一エンドポイント拡張 vs 専用エンドポイント)、think-strip のストリーミング対応方針、部分失敗時の扱い、プロンプトチューニングの範囲。

## Decision

### 1. ストリーミング専用エンドポイントを新設する

`POST /api/v1/articles/{id}/summary/stream` を新設し、既存の `POST /api/v1/articles/{id}/summary`(一括JSON)はそのまま残す。両方とも冪等(既に要約済みなら新規生成しない)。

- **却下した代替案1**: 既存エンドポイントに `Accept` ヘッダでの content negotiation を持たせる案。1エンドポイント1応答形式の単純さを優先し、専用エンドポイントの新設を選んだ(fulltext/summarize の「1機能1エンドポイント」規約が既に確立しているため、応答形式ごとにエンドポイントを分ける方が実装・テストとも素直)。
- **却下した代替案2**: 既存エンドポイントを廃止して常時SSE化する案。E2E(Hurl)の一括JSONアサーションの書き換えコストに見合わないと判断した。
- 新規生成時は think 除去済みの断片を `event: delta` で逐次送信し、生成完了後 `event: done` で保存済みSummaryを送る。既に要約済みの場合も同じSSE形式で応答するが、`delta` は1回(全文)だけ送ってから `done` を送る — クライアント側の実装をストリーミング/非ストリーミングで分岐させないため。
- SSEヘッダ送出(`w.WriteHeader(200)`)後はHTTPステータスを変更できないため、生成途中の失敗は `event: error`(`{"error": ..., "status": ...}`)で伝える。HTTPステータス自体は200のまま。

### 2. think-strip は「先頭バッファ方式」でストリーミング対応する

`core/internal/summarize/summarize.go` に `thinkStreamStripper` を追加した。チャンクが届くたびに次の状態機械で処理する:

- **undecided**: `<think>` の開始タグ長(7バイト)に達するまで内部バッファし、何もクライアントへ流さない。7バイト到達時にプレフィックス一致を判定する。
- **insideThink**: `<think>` が見つかった場合、`</think>` が見つかるまで一切クライアントへ流さない。
- **passthrough**: `<think>` が無い(または `</think>` を通過した)後は、以降のチャンクを即座にそのまま流す。

`<think>` が閉じずに応答が終わった場合(ADR00014 §5 が想定する既知の回帰と同じ状況)、`passthrough` に一度も遷移しないため **クライアントには何も表示されていない状態のまま** SSE `error` イベントを送れる — 「中途半端なCoTを見せてしまう」という失敗モードが構造的に発生しない。

最終的な保存判定(DB に保存する `summaryText`・`think_stripped` フラグ)は、ストリーミングの状態機械ではなく **常に既存の `stripThink(完全な生テキスト)` を正とする**。ストリーミング側の状態機械はリアルタイム表示専用の補助ロジックであり、二重の判定ロジックを持たせない。

- **却下した代替案**: 生トークンをそのままストリームし、think除去をクライアント(Web側)に担わせる案。ドメインロジックがクライアントに漏れ、clean-architecture の層分離が崩れるため却下。

### 3. ストリーミング中の失敗は部分テキストを保存しない

接続断・LLMエラーが発生した場合、それまでにストリームした部分テキストは `article_summaries` に一切保存しない。`enrichment_attempts` に `kind='summary', outcome='failed'` のみ記録する。ADR00014 §7(リトライ・サーキットブレーカー無し、単発呼び出し)の方針をストリーミングでも踏襲した。

- **却下した代替案**: 部分テキストを `partial` フラグ付きで保存し再開の材料にする案。単一ユーザー・ローカルLLMという運用規模でミニマリズムに反すると判断し却下(再要約は読者の再クリックで十分)。

### 4. internal/llm への切り出しはしない

ADR00014 §1 と同じ YAGNI 判断を維持した。`Completer` インターフェースに `CompleteStream` メソッドを追加するに留め、consumer は summarize パッケージのみという状況は変わっていない。

### 5. プロンプトをA/Bテストで改善する

`eval/` の generate コマンドはモデル間比較のみを想定していたため、`--prompt`(プロンプトファイル差し替え)と `--label`(出力レコードの `model_key` 上書き)を追加し、**同一モデル・異なるプロンプト**でのA/B比較を既存の pairs/judge/score ハーネスに乗せられるようにした。

比較対象:

- **variant A**(現行本番プロンプト): 「400字程度、背景・具体的な内容・結論を含め、一言二言で終わらせない」(ADR00014 で確定)。この機会に `eval/prompts/summarize_ja.txt` を古い「3〜5文」版からこちらに同期させた。
- **variant B**(改善案): 「(1) 背景・前提となる状況、(2) 記事固有の具体的な事実・数値・固有名詞、(3) 記事が示す結論または帰結」を明示的な列挙要件にし、「一般論や紋切り型の言い回しで字数を埋めない」という否定制約を追加。

`qwen35-4b` で両バリアントを凍結済み30記事に対して生成し(`promptab-a` / `promptab-b`)、ブラインド化ペア化・位置スワップ両順込みで判定した。

**審判はClaude(Sonnet 5)単独**で行った。当初はADR00007に倣い gpt-oss-20b(ローカル)との二重審判を実施する計画で、実際に120件中一部まで実行したが、ユーザーから「gpt-oss-20bの審判に時間がかかりすぎている、Sonnet 5の判定だけでよい」との指示があり、ローカル審判は打ち切ってClaude単独の結果を採用した。今回は**モデル間比較ではなくプロンプト間比較**であり、ADR00007のモデル選定ほど大きな不可逆性を持たないため、単独審判での簡略化は許容できると判断した。

結果(30ペア): **variant B 18勝 - variant A 6敗 - 6分、sign-test p=0.023**(tie除外の両側符号検定、ADR00007と同一プロトコル)。決着ペア(24ペア)中のB勝率75%。観点別では faithfulness(忠実性・捏造抑制)と conciseness(簡潔さ・冗長排除)で B が明確に優勢、coverage(網羅性)はやや優勢、instruction(指示遵守)はほぼ差なし(tie多数)。position_consistency = 1.0(位置バイアスの疑いなし)。

variant B を本番プロンプト(`core/internal/summarize/client.go`)と `eval/prompts/summarize_ja.txt` の両方に反映した。候補ファイル `eval/prompts/summarize_ja_v2.txt` は本番採用後に削除した。

## Consequences

### Pros

- ストリーミングにより読者は生成完了を待たず要約が表示され始める(体感レイテンシの改善)
- think-strip の先頭バッファ方式により、ストリーミングでも「中途半端なCoTを見せる」失敗モードが構造的に発生しない
- eval/ のプロンプト差し替え機能により、今後もプロンプト変更のたびに定量A/Bを回せる土台ができた(モデル選定用ハーネスをそのまま流用でき、新規ツール不要)
- eval/prompts/summarize_ja.txt と本番プロンプトの乖離を解消し、以後の評価が本番の実態を反映するようになった
- 新規エンドポイントのみの追加で、既存の一括JSON API・E2Eシナリオは無改修のまま動作する

### Cons / Tradeoffs

- **審判がClaude単独**であり、ADR00007のような「候補と別族の最強単一審判 + Claude」の二重審判プロトコルを完遂していない。ローカル審判(gpt-oss-20b)との一致率という形でのメタ評価(審判自体の信頼性検証)が今回は得られていない
- think-strip の先頭バッファは `len("<think>")` = 7バイト固定。将来 think タグの綴りが変わったり、別の予告トークンを使うモデルに切り替わった場合は定数を見直す必要がある
- ストリーミング用の `thinkStreamStripper` と非ストリーミング用の `stripThink` は別実装であり(保存判定は後者に一本化したが)、ロジックが完全に一致していることをテストで担保する以外に構造的な保証はない
- `promptab-*` の生成・判定結果(eval/results/pairs, judgments, generations)は一度きりのA/Bであり、継続的なプロンプト回帰テストの仕組みにはなっていない

## Revisit Triggers

- 実運用で「要約が中途半端に切れる」「thinkタグが漏れて見える」といった報告があれば、`thinkStreamStripper` のバッファ長・状態遷移を再検証する
- llama.cpp のバージョン更新(ADR00006)時、SSEチャンク形式(`delta.content` の名称・`[DONE]` 終端)が変わっていないか実機確認する
- 次にプロンプトを変更する際は、今回作った `--prompt`/`--label` を使い、可能であればgpt-oss-20bとの二重審判(ADR00007と同一プロトコル)をやり直して審判の一致率を確認する
- タグ抽出・埋め込み等、2人目のLLM消費者が要約以外で登場するタイミングで `internal/llm` への切り出しを再評価する(ADR00014 から持ち越しの Revisit Trigger と同じ)
- 400字/8〜12文という分量目標自体への読者フィードバックが蓄積したら、分量そのもののA/Bも検討する
