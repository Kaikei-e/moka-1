# 記事一覧 API を offset ベースからカーソルベース(keyset)ページングへ変更する

## Context

[[ADR00011]] 時点の `GET /api/v1/articles` は `limit`/`offset` ページングだった。offset 方式には構造的な問題が二つある:

1. **深いページで線形に遅くなる**: `OFFSET n` は先頭から n 行を読んで捨てる。記事テーブルは常駐エージェントの取得ループで単調に増え続けるため、時間とともに悪化する種類のコスト
2. **ページ境界がずれる**: ページ間に新着記事が挿入されると、offset 基準では同じ記事が二度出る/取りこぼす。RSS リーダーは「読んでいる間に新着が来る」のが常態なので、これは理論上の問題ではない

初期実装のうちに直す方が安い(消費者は moka-web の BFF ひとつ、`/api/v1` はまだ外部契約を持たない)。

## Decision

**keyset(カーソル)ページングに全面移行する。offset は残さない。**

- カーソルは並びキー `(published_at DESC NULLS LAST, id DESC)` の「最後に返した行」を base64url で不透明にエンコードした文字列。形式(`RFC3339Nano|id`)はサーバー実装の私事であり、クライアントには契約させない
- レスポンスは `{"articles": [...], "next_cursor": "..." | null}`。キーは終端でも常に返し(契約の安定)、`null` が終端を意味する。満杯ページのみ `next_cursor` を発行する
- `published_at IS NULL` の行(NULLS LAST の末尾領域)もカーソルで表現できる — タイムスタンプ部が空のカーソルは NULL 領域内を指し、以後は `id` だけで単調に進む
- 壊れたカーソルは `ErrInvalidCursor` に畳んで 400(内部形式をエラーメッセージで漏らさない)
- インデックスを `(published_at DESC)` 単独から `(published_at DESC NULLS LAST, id DESC)` の複合に置き換える。旧インデックスは DESC NULLS FIRST でありこの並びに使えない — **インデックスを揃えない keyset は offset を消しただけでスケールしない**

検討した代替案:

1. **offset を維持**(却下)— 上記 2 問題が時間とともに顕在化する。直すなら消費者が BFF ひとつの今
2. **offset と cursor の併存**(却下)— 経路が 2 本になり、テスト・インデックス・ドキュメントの守備範囲が倍になる。ミニマリズムに反する
3. **カーソルを平文クエリパラメータ(`before_published_at=&before_id=`)にする**(却下)— 並びキーの構成がクライアント契約になり、将来ソート順を変えるとき(例: ハイライトスコア順)に破壊的変更になる。不透明カーソルなら中身の差し替えが自由
4. **created_at / id 単独キーで並べる**(却下)— 読者にとっての「新しい順」は published_at。取得順(created_at)とは一致しない

## Consequences

### Pros

- ページ取得コストが深さに依存しない(複合インデックスのレンジスキャン + LIMIT)
- ページ間の新着挿入で重複・取りこぼしが起きない(位置でなく値を基準に「続き」を引くため)
- 不透明カーソルにより、並びキーの変更(将来のスコア順など)がクライアント非破壊でできる

### Cons / Tradeoffs

- 「n ページ目へジャンプ」ができない(keyset の原理的制約)。読書 UI は前から順に読む前提なので現時点で必要ない
- `next_cursor` の発行規則が「満杯ページのみ」なので、総件数がちょうど limit の倍数のとき最後に空ページを 1 回踏む(実害なし、COUNT を避ける定石)
- published_at 同値の行は id でタイブレークするが、時計が逆行するようなフィード(published_at が後から変わる上流)には追従しない — 記事は immutable(ADR00001)なのでこれは仕様
- `atlas migrate lint` は dev-database 未設定のため未実施(DROP INDEX を含むマイグレーションなので、CI に lint を組み込む際に要確認)

## Revisit Triggers

- ハイライト/スコア順など published_at 以外の並びを一覧 API に足すとき — カーソルの中身を並びごとに定義し直す(不透明形式なので API 契約は保てる)
- 「合計件数」や「ページ n/m」表示が UI 要件になったとき — keyset と COUNT の併用コストを再評価する
- UI が実際にカーソルを辿る実装(「さらに読む」等)を入れるとき — BFF に next_cursor の伝搬を足す(現状は 50 件固定で辿っていない)
