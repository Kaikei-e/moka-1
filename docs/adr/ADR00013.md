# 記事全文の取り寄せを go-trafilatura + イベントテーブル + 冪等 API で実装する

## Context

Web UI の記事一覧から選んだ記事1本だけ、読者の明示操作で外部サイトから全文を連れてくる機能が必要になった。前提となる制約:

- `articles.content` は RSS フィードの `content`(無ければ `description`)であり、多くのフィードでは概要止まり(tenets §3.2 の正規化ステップの成果物であり、本文抽出はこれまで未実装だった)
- tenets §7 は本文抽出ライブラリの選定根拠として ScrapingHub ベンチの go-trafilatura F1=0.960(Python 版 0.958 とほぼ同点・Go の方が高速)を既に引用しており、Go 実装であること自体は既定路線だった
- articles の URL は RSS フィードの `link`/`guid` 由来で、フィード登録時の SSRF 検証([[ADR00010]])の対象外(フィード URL とは別の値)。全文取り寄せで新たに外部リクエストを打つ以上、同じ SSRF ガードが要る
- [[ADR00002]] の immutable データモデル(リソースは INSERT-only の別テーブルとして分離し、`articles` に enrichment_status 相当のカラムを足さない)がある
- [[ADR00011]] は「クライアント起点の再取得は `+server.ts` が要る」と明記して未実装のまま残していた。今回がその最初の実例

## Decision

### 1. 抽出ライブラリは `github.com/markusmobius/go-trafilatura`(v1.12.2)

独立ベンチ(ScrapingHub `article-extraction-benchmark`、2026-05 更新)で F1=0.960 と最高精度。フォールバック抽出(`EnableFallback: true`)で go-readability / go-domdistiller に自動委譲する。

検討した代替案:

- **`codeberg.org/readeck/go-readability/v2`**(却下、次点として記録): F1=0.947 とやや劣るが依存が軽い(WASM ランタイム無し)。依存重量を最優先するなら次点候補
- **`go-shiori/go-readability` 単体**(却下): 2025-12 にアーカイブ済み・readeck フォークへの移行が公式に案内されており新規採用しない
- **プレーンテキスト抽出(`ExtractResult.ContentText`)をそのまま使う**(却下、実装後に判明): `ContentText` は段落間の改行を保持せず全文を1行に平坦化する。読書カラムは段落のリズムが設計原則(§3.2 行間 2.05・測度 38 字)なので、代わりに `ExtractResult.ContentNode` を `go-shiori/dom.OuterHTML` で段落構造を保った HTML(`<p>` タグ区切り)にシリアライズし、フィード由来の `articles.content` と同じ形にして返す。フロントエンドは既存の `toParagraphs`(タグ除去 + XSS 安全化)をそのまま再利用できる。見出し(`h1`〜`h6`)は読書ビューのタイトルと重複するため抽出結果から除去する

### 2. 保存先は `article_fulltexts` イベントテーブル(INSERT-only)

`articles` にカラムを足さない。`(article_id, fetched_at DESC)` の最新行が有効(ADR00002 と同型)。

検討した代替案:

- **`articles.content` を UPDATE で上書き**(却下): immutable モデル(ADR00002)に反し、「RSS 由来の概要」という `content` の意味も壊れる
- **都度取得・DB に残さない**(却下): 再訪のたびに外部サイトへ再アクセスすることになり、tenets §3.2 の外部リクエスト間隔 ≥5s の精神(politeness)と衝突する

### 3. 冪等セマンティクス: 保存済みなら再取得しない(新規 201 / 既存 200)

[[ADR00010]] のフィード登録(冪等・409 を返さない)と同じ思想。取得失敗はイベントとして記録しない — フィード取得の背景バッチ処理と異なり、取り寄せは**ユーザー起点の同期処理**で、失敗はその場で HTTP エラーとして返せば足りる(`enrichment_attempts` のような再処理キューは不要)。

### 4. SSRF 検証は `feed.URLValidator` を再利用

`internal/fulltext` は `internal/feed` の `*feed.URLValidator` を `Validator` interface 経由でそのまま注入する(具象の共有)。SSRF ガードの実装を二重に持たない。エラー分類(`ErrInvalidURL` / `ErrPrivateHost` / `ErrUpstreamFetch`)も `feed` パッケージの sentinel をそのまま re-export し、`httpapi` 側のステータス写像([[ADR00010]] と同じ 400/422/502 パターン)を揃える。

### 5. moka-web は `+server.ts` 経由のクライアント起点フェッチ([[ADR00011]] の実例化)

読書ビューの「全文を取り寄せる」ボタンが `POST /articles/{id}/fulltext`(moka-web 内のサーバールート)を叩き、そのルートが `lib/server/core.ts` 経由で moka-core を呼ぶ。ブラウザは moka-core を直接叩かない([[ADR00011]] の境界を維持)。

### 6. 用語: 「全文」「取り寄せ」を「濃縮」と別概念にする

`CONTEXT.md` に登録済み。濃縮(enrichment)は LLM 生成物で常に `fujinezu` 面、取り寄せは記事の声そのもの(外部サイト由来、`kumoi` 地・明朝)。ボタン文言は「全文を取り寄せる」(コピー規則 §7.2: 動詞から)。

### 7. 副次的決定: E2E 実行専用の `e2e-db` / `e2e-migrate` を追加

本題の実装ではないが、この機能の E2E(Hurl)を回す過程で、`compose.e2e.yaml` オーバーレイが独立 DB を持たず本番の `db` サービス/`db-data` ボリュームを共有していたことが判明した(過去のセッションで `down -v` せずに E2E を回し、本番の開発用 DB に E2E フィクスチャの feed が残留していた)。`compose.e2e.yaml` に `e2e-db`(専用ボリューム `e2e-db-data`)と `e2e-migrate` を追加し、`moka-core` の `DATABASE_URL` を E2E 実行時だけそちらに向けた。`e2e-fixtures` と同じく E2E 限定サービスなので常駐サービス 5 上限(tenets §2-3)の勘定外。

## Consequences

### Pros

- 独立ベンチで最高精度の抽出器を採用しつつ、段落構造を保つことでフロントエンドの実装(`toParagraphs`)を増やさずに済んだ
- SSRF ガード・エラー分類・冪等パターンをフィード登録(ADR00010)と揃えたことで、読み手が同じ語彙で両方を理解できる
- immutable モデル(ADR00002)を崩さずに機能追加できた
- E2E が本番の開発用データを二度と汚さない構成になった(副次的だが再発防止として重要)

### Cons / Tradeoffs

- go-trafilatura は `wasilibs/go-re2`(WASM ランタイム経由の正規表現エンジン)を間接依存に持ち、バイナリサイズが数 MB 増える。依存の軽さを優先するなら次点の readeck/go-readability v2 へ切り替える余地を残す
- 取得失敗を記録しないため、「どの記事の取り寄せが何度失敗したか」の統計は持てない(ユーザー起点の同期処理である前提が崩れたら再検討)
- `e2e-migrate` の追加により E2E 起動シーケンスが 1 ステップ増えた(本番の起動シーケンスには影響しない)

## Revisit Triggers

- 取り寄せた全文を要約の入力に使う展開になったら、「取り寄せ」と「濃縮」の概念境界(CONTEXT.md)を再確認する
- バイナリサイズや起動メモリが問題になったら、go-trafilatura → readeck/go-readability v2(F1=0.947・依存軽量)への切り替えを検討する
- 取り寄せの再試行 UX(force オプション等)が要求されたら、現在の「保存済みなら常に既存行を返す」冪等セマンティクスを見直す
- E2E 実行頻度が上がり `e2e-migrate` の起動コストが無視できなくなったら、e2e-db を都度作り直さず使い回す設計を検討する
