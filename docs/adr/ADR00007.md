# 実測A/Bに基づきLLM編成を確定する(高速パスをQwen3.5-4Bへ、スワップ枠をQwen3.6-35Bに一本化)

## Context

ADR00006 で 6 つの GGUF を pin し速度は実測済みだったが、品質は未計測だった(ADR00006「量子化の品質影響は未計測」、tenets §10)。未決は (1) 高速パスの品質A/B、(2) 集約層の常駐 vs スワップ、(3) スワップ枠の Qwen3.5 vs 3.6、(4) タグ抽出の構造化出力品質。tenets は「採用判断は必ず自前データ(日本語記事)でのA/Bを経る」ことを要求する。

`eval/` に選定ハーネスを新設し、公開RSSから凍結した30記事(JPテック20+JP一般5+EN5、`eval/data/articles/manifest.json`)で計測した。品質判定は最新研究に基づくプロトコル(根拠と出典: `eval/results/research/local-judge.json`, `moa-theory.json`):

- ブラインド化ペア + **位置スワップ両順一致のみ勝ち**(位置バイアスで判定の~2割が反転するため)
- 審判は候補と別族の最強単一審判(gpt-oss-20b)+ Claude エージェント審判の二重体制。弱い異種審判の多数決は組まない(エラー相関で実効票が増えない)
- 判定は自由記述CoT→ `[[A]]/[[B]]/[[tie]]` タグ抽出(json_schema で縛ると推論が10-15%劣化する報告)
- 観点レンズ: 忠実性/網羅性/簡潔さ/指示遵守(「日本語の自然さ」は人手相関が低く不使用)
- 集計は tie 除外の両側符号検定。採用閾値は p<0.05 かつ決着ペア勝率≥60%

主要実測(詳細: `eval/results/summary/decision-matrix.json`):

| 項目 | 実測 |
|---|---|
| D1 要約品質 | **Qwen3.5-4B 18勝-7敗-5分 vs LFM2.5、p=0.043、勝率72%**(両審判とも同方向、審判間一致0.60) |
| D1 実効レイテンシ | Qwen3.5-4B(no-think)**6.7s/記事** vs LFM2.5 **14.7s**(常時CoTが生成トークンの82%を占有、素の67 t/sが実効10.2 t/sに落ちる) |
| D1 言語逸脱 | LFM2.5 は30本中**7本を英語で要約**(指示違反)。Qwen3.5-4B は 0 本 |
| D5 タグ | 品質同格(p=0.69)、両者 json_schema パース成功率 100% |
| D3 リキャップ品質 | Qwen3.6-35B が全対戦勝ち(vs 3.5: 2-1、vs gemma-4-26B: 3-0 両審判一致、vs gpt-oss: 3-0)。n=3週で有意には届かないが方向一貫 |
| D2 ロード時間 | gpt-oss ウォーム4-5s / 35B級は毎回38-42s(ホストRAM16GBでページキャッシュ不成立) |
| D2 常駐可否 | KVデフォルト確保では**ホストRAM起因のOOMを実測**(2026-07-03、kernel oom-kill)。`-c 8192` に制限した再プローブでは5モデル同時常駐が安定成立(**VRAM 45.9/48.0GiB・GTTスピルほぼゼロ・全常駐下でも高速パス劣化ゼロ/35B -1%**) |

## Decision

確定編成(tenets §3.3 を本ADRで更新):

| 役割 | モデル | 変更 |
|---|---|---|
| 高速パス(要約・タグ・翻訳) | **unsloth/Qwen3.5-4B-GGUF:Q4_K_M**、`chat_template_kwargs: {"enable_thinking": false}`、温度0.7/top_p 0.8/top_k 20 | LFM2.5 から交代 |
| 提案層 | LFM2.5-8B-A1B(専任に降格)/ gemma-4-E4B / Qwen3.5-4B(兼務) | LFM2.5は網羅性レンズでは勝っており提案役に残す |
| 集約層(審判・統合・RAG) | ggml-org/gpt-oss-20b-GGUF(MXFP4)常駐 | 維持 |
| スワップ枠(週次リキャップ等の品質最優先バッチ) | **unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL**、リキャップは max_tokens 8192(CoTが4096超の実測) | Qwen3.5-35B を破り一本化 |

検討した代替案と却下理由:

- **高速パス LFM2.5 維持**: 却下。品質で有意に負け(p=0.043)、常時CoTで実効2.2倍遅く、日本語指示下で23%が英語出力。網羅性単独では勝るが、忠実性(3-14)と簡潔さ(1-11)の大差を覆せない。
- **スワップ枠 gemma-4-26B-A4B**(google公式QAT q4_0、13.9GB): 却下(記録として残す)。tg 28.0 t/s(+19%)・VRAM 19.9GiB と速度/サイズは最良だったが、リキャップ品質で Qwen3.6 に 0-3(両審判一致)。
- **スワップ枠 Qwen3.5-35B 維持**: 却下。3.6 に品質で負け(2-1+Claude単独2-0-1)、ADR00004 の採用条件「品質A/Bのみ」を 3.6 が充足。敗者GGUF(20.7GB)は ADR00006 の指示に従い削除対象(実削除はユーザー確認後)。
- **集約層の35B常駐化(「両方常駐」)**: 却下(ただし技術的には成立を実証)。`-c 8192` なら5モデル同時常駐45.9/48.0GiB・性能劣化なしで動くが、残ヘッドルーム約2GiBはデスクトップ併用(ベースラインでGPU使用~7GiBの実測あり)に対して薄く、KV設定を誤ると即OOMする脆い構成になる。週次バッチに毎回38-42sのロードは無視できるため、堅牢なスワップ運用を採用。
- **集約層を35Bスワップに置換(gpt-oss廃止)**: 却下。gpt-oss はウォームロード4-5秒で審判・RAG最終回答のインタラクティブ用途に必須。リキャップ品質は 3.6 が上のため、**リキャップだけスワップで格上げ**する分業に。

## Consequences

### Pros

- 高速パスの体感が記事あたり中央値 14.7s→6.7s に半減し、英語混入が消える
- 週次リキャップの品質が全対戦最強の Qwen3.6-35B に(262Kコンテキストの余裕も維持)
- 判定プロトコル(ブラインド+位置スワップ+二重審判、位置一貫性0.89-0.96)が `eval/` に再利用可能な形で残った
- ディスク: gemma-4-26B は不採用だが実測済み記録があり、Qwen3.5-35B 削除で 20.7GB 回収可能

### Cons / Tradeoffs

- D3 は n=3 週で統計的有意に届かない(方向は全対戦一貫)。リキャップ実運用データが貯まったら再検証
- LFM2.5 の網羅性優位を高速パスから失う(提案層で活かす)
- Qwen3.5-4B の TTFT は LFM2.5 より遅い(1563ms vs 630ms、pp 426 vs 949 t/s)— 長文prefill主体の用途では逆転しうる
- 審判間一致率 0.47〜0.60 は中程度。僅差判定は tie に寄る保守的プロトコルであることを前提に読むこと
- ハイライト3異種提案層は Self-MoA 研究(arXiv 2502.00674: 品質>多様性)と緊張関係にある。M3 で「最良提案者×3サンプル vs 3異種」のA/Bを行うまで現構成を仮置き

## Revisit Triggers

- リキャップ/要約の実運用ログで品質不満が観測された(モデル系譜は `model_meta` に記録済みで A/B 再実行可能)
- LiquidAI LFM3(AMD最適化を予告)や Qwen3.5-4B 後継の公式GGUFが出た
- GPT-OSS-Swallow-20B の公式/準公式GGUFが出た(集約層の最有力交代候補)
- llama.cpp の pin 更新時(全モデル `llama bench` 再実測とセット、ADR00006)
- ホストRAM増設またはカーネル設定移行(carve-out最小+GTT大盛り、`eval/results/research/uma-vram.json` 提案 (d))で「35B常駐」の前提が変わった
- スワップ枠の Q5_K_XL 昇格検討(KLD -42%/decode -10〜15% の実測報告あり)
