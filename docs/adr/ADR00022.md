# ハイブリッド検索のテキスト側を pg_trgm で実装する

## Context

Tenet 4(単一DB)は当初「全文検索は PostgreSQL の FTS + pgvector ハイブリッドで賄う」と書いていた。しかし素の tsvector は日本語を分かち書きできない — `simple` 設定では文全体がほぼ1トークンになり、日本語主体のフィードでは検索として実質機能しない。「PostgreSQL の FTS」という字面と日本語の現実が衝突していた。

M2(埋め込み+ハイブリッド検索+問い返し)の実装計画(2026-07-19 のグリルセッション)で、ハイブリッド検索のテキスト側の実体を決める必要が生じた。ベクトル側は [[ADR00008]] で Qwen3-Embedding-0.6B + pgvector `vector(1024)`(cosine)に確定済み。検索UIも M2 に含めることが同セッションで決まっており、テキスト側を先送りすると tenets の「ハイブリッド」が未達のまま残る。

## Decision

**pg_trgm(標準 contrib)を採用し、ハイブリッド検索のテキスト側とする。**

- 現行の DB イメージ `pgvector/pgvector:0.8.4-pg18-trixie` に同梱の標準 contrib であり、`CREATE EXTENSION pg_trgm` だけで使える(イメージ変更・pin 変更なし)
- trigram は文字 n-gram ベースなので形態素解析(分かち書き)を必要とせず、日本語でも実用精度が出る
- articles(title, content)への GIN trgm インデックスで類似度検索を支える
- 融合は pgvector cosine 近傍との RRF(Reciprocal Rank Fusion, k=60)を既定とし、重み・品質は eval/ の retrieval ハーネス(recall@k / MRR — [[ADR00008]] で構築済み)で実測調整する
- Tenet 4 の文言は 2026-07-19 付で「pg_trgm + pgvector ハイブリッド」に更新済み(`docs/tenets/moka-tenets.md` §2-4)

検討した代替案と却下理由:

- **PGroonga**: 日本語FTSとしては最強(形態素・N-gram 両対応)だが、pgvector との同居にカスタム DB イメージのビルドが必要になる。イメージ pin 原則(公式イメージ + digest pin)と One-kick の整合コストが増え、OSS としてクローンする利用者にもイメージビルドの負担を強いる。現段階の個人規模 RSS 検索に対して過剰
- **tsvector simple の強行**: tenets の旧文言の字面どおりだが、英語フィードのみ有効の中途半端な実装になる。日本語主体という前提に反する
- **M2 をベクトル単独で出荷しテキスト側は後送り**: 埋め込み検索は自然文クエリに強い一方、固有名詞・キーワードの完全一致系クエリで取りこぼす。tenets が約束する「ハイブリッド」未達が残り、後送りの回収時期も不明瞭

## Consequences

### Pros

- イメージ変更ゼロ・拡張1つで日本語テキスト検索が成立する(単一DB原則・イメージ pin 原則・One-kick をすべて維持)
- キーワード型クエリ(pg_trgm)と自然文クエリ(pgvector)を相互補完する構成になり、問い返しの retrieval 層と検索UIを同じ基盤で支えられる
- RRF の重み調整が eval/ の既存 retrieval ハーネスでそのまま実測できる

### Cons / Tradeoffs

- trigram の原理上、2文字クエリ(「東京」「AI」等)に弱い — 3-gram が作れず類似度が粗くなる。短いクエリはベクトル側の寄与でカバーする前提
- 形態素ベースの検索(語境界の正確な一致、活用の吸収)は提供できない。PGroonga 比で日本語精度の上限は低い自覚を持つ
- GIN trgm インデックスは記事本文に対してサイズが嵩む傾向がある(現規模では許容、肥大したら対象カラムを見直す)
- スキーマ変更(`CREATE EXTENSION` + インデックス追加)は本ADRの範囲外のフォローアップ実装とする([[ADR00008]] と同じ流儀 — 本ADRは決定の記録まで)

## Revisit Triggers

- 実運用で検索の外れ(取りこぼし・ノイズ)が体感された — その時は PGroonga 再評価をカスタム DB イメージのコスト(ビルド・pin・OSS 利用者負担)を払う判断とセットで行う
- 2文字クエリの弱さが検索UIの体感を損ねている実例が蓄積した(bigram 系対策や instruction prefix 側の工夫を含めて再検討)
- 記事数が数千件規模に達し、eval/ の実データ再実測で RRF 融合の precision/recall が劣化した
- DB イメージ側の事情で pg_trgm の同梱が変わった(可能性は低いが pin 更新時に確認)
