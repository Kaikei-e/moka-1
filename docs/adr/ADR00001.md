# DBマイグレーション管理にAtlasを採用する

## Context

moka-1 の設計原則(tenets §2)は「One-kick」と「サービス数上限 5」を掲げ、スキーマ管理の当初案は「`embed.FS` に SQL を持ち moka-core 起動時に自動適用」だった(tenets §4)。しかし:

- 手書きの逐次適用機構は **lint / plan / ドリフト検出を持たない**。pgvector の HNSW インデックスや FTS の式インデックスなど、書き方を誤るとテーブルロックで常駐エージェントを止める変更を、無検査で本番(=唯一の個人環境)に流すことになる
- スキーマは 4 テーブル+αと小さいが、`enrichment_status` の遷移や embedding 次元変更(モデル交代時)など、**破壊的変更が予見される**(tenets §8-1: モデル最終選定が未決)
- Alt で Atlas(`atlas migrate hash` / `atlas migrate apply`)を既に運用しており、習熟コストは実質ゼロ

ミニマリズム原則の下では自前 embed.FS が「最小」だが、**マイグレーションは失敗したときの復旧コストが最も高い領域**であり、ここだけは複雑度を意識的に引き受ける価値があると判断した。

## Decision

**Atlas の versioned migrations を採用する。**

1. **ディレクトリ**: リポジトリ直下に `db/` を新設
   - `db/schema.sql` — 望ましいスキーマの単一ソース(declarative)
   - `db/migrations/` — `atlas migrate diff` が生成する versioned SQL + `atlas.sum`
   - `db/atlas.hcl` — env 定義(local / test)
2. **開発フロー**: `schema.sql` を編集 → `atlas migrate diff` で差分 SQL 生成 → `atlas migrate lint` で破壊的変更を検査 → commit。手書き SQL を直接 `migrations/` に足す場合も `atlas migrate hash` で `atlas.sum` を更新する(Alt と同じ運用)
3. **適用は compose 内の one-shot ジョブ**: `arigaio/atlas` イメージ(バージョン pin)の `migrate` サービスが `atlas migrate apply` を実行して exit 0 する。`moka-core` は `depends_on: migrate: condition: service_completed_successfully` で待つ。**One-kick は維持され、手動手順は増えない**
4. **サービス数上限 5 の解釈を明確化**: 上限は「常駐サービス」を数える。走って終了する one-shot ジョブは数に含めない(tenets §2-3 に注記)
5. **moka-core は自前マイグレーション機構を持たない**: 起動時にスキーマバージョン検査もしない(compose の依存順序が保証する)。`embed.FS` 案は廃止

**検討した代替案と却下理由**:

- **`embed.FS` + 自前逐次適用(当初案)**: 依存ゼロで最小だが、lint / plan / ドリフト検出なし。破壊的変更を無検査で唯一の環境に流すリスクが許容できない
- **golang-migrate**: Go 親和性は高いが up/down の手書き管理のみで、schema-as-code(diff 生成)も lint もない。Atlas に対する優位が「軽い」しかない
- **declarative 適用(`atlas schema apply` を起動時に直接実行)**: 手数は最小だが、適用される SQL が実行時の DB 状態に依存して決まり、versioned migrations に比べ再現性・監査性で劣る。個人運用でも「何が流れたか」が git に残る方を取る

## Consequences

### Pros

- 破壊的変更の事前検出(`atlas migrate lint`)と適用計画の可視化
- スキーマの単一ソース(`schema.sql`)と適用履歴(`migrations/`)が両方 git に残る
- Alt と同一ツールで、運用知識・トラブル対応が共有できる
- one-shot ジョブ方式により moka-core イメージに atlas バイナリを同梱しなくてよい

### Cons / Tradeoffs

- 外部ツール依存が 1 つ増える(イメージ pin と更新追随が必要)
- ミニマリズム原則の明示的な例外第 1 号(前例として引用されるリスク。「マイグレーションは復旧コスト最大領域」という限定理由ごと引用すること)
- Atlas の一部機能(高度な lint アナライザ等)はログイン必須の Pro 機能。コミュニティ版の範囲で運用する

## Revisit Triggers

- Atlas のライセンス・配布形態が変わり、コミュニティ版で versioned migrations + lint が使えなくなったら(golang-migrate + 手動レビューへの縮退を評価)
- one-shot ジョブが compose 起動の信頼性問題(タイミング・リトライ)を反復して起こしたら(atlasexec による moka-core 内蔵実行を評価)
- スキーマ変更が年数回以下で安定したら(過剰装備になっていないかの見直し)
