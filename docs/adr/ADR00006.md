# 事前配置する GGUF の配布元と量子化形式を固定する

## Context

tenets §3.3 の LLM 編成はモデル名とサイズ感(例: Qwen3.5-4B ~2.6GB)までしか定めておらず、**どの配布元のどの量子化を使うか**が未決だった。同一モデルでも配布元・量子化で品質とサイズが大きく変わるため、eval/ での A/B を始める前に「何を比較するか」を固定する必要がある。Web 調査(2026-07-02)と Vulkan 実測で決定した。

ダウンロードは `llama download -hf <repo>:<quant>`(llm コンテナ内)で行い、`LLAMA_CACHE=/models` により HF hub 形式で `models/` に永続化される(One-kick: 高速パスは llm サービス起動時の自動 DL)。

## Decision

**以下の 6 本に固定する。**Vulkan 実測(RADV gfx1150 / llama.cpp b9859 / `-ngl 99` / `llama bench -r 2`)を選定の証跡として残す:

| 役割 | 配布元:量子化 | サイズ | pp512 | tg128 |
|---|---|---|---|---|
| 高速パス(常駐) | `LiquidAI/LFM2.5-8B-A1B-GGUF:Q4_K_M`(公式) | 4.8G | 1067.9 | **69.5** |
| 提案層A(常駐) | `unsloth/Qwen3.5-4B-GGUF:Q4_K_M` | 2.5G | 568.1 | 25.1 |
| 提案層B(常駐) | `unsloth/gemma-4-E4B-it-qat-GGUF:UD-Q4_K_XL` | 3.9G | 576.6 | 27.6 |
| 集約層(常駐) | `ggml-org/gpt-oss-20b-GGUF`(MXFP4) | 11.3G | 519.4 | 30.1 |
| 交代枠・第一候補 | `unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL` | 20.8G | 343.4 | 23.3 |
| 交代枠・比較用 | `unsloth/Qwen3.5-35B-A3B-GGUF:UD-Q4_K_XL` | 20.7G | 350.2 | 23.4 |

選定理由:

- **LFM2.5 は Liquid AI 公式 GGUF**。tg 69.5 t/s は draft_0001 の仮説レンジ(60–75)に的中。`-np 3` バッチで合計 87.5 t/s(1 ストリーム ~29 t/s)も実測済みで、MoA 提案層の 3 ペルソナ同時デコードが成立する
- **Gemma4 E4B は QAT 版**(Google の Quantization-Aware Training 重み + unsloth Dynamic)。非 QAT の Q4_K_M(4.64GiB)より小さい 3.93GiB で bf16 相当品質を謳う。「4bit にするなら QAT がある方を選ぶ」を原則化する
- **gpt-oss-20b は llama.cpp 公式 org(ggml-org)の MXFP4**。OpenAI が MXFP4 でネイティブ学習したモデルなので、これ以外の再量子化を選ぶ理由がない
- **35B 級(交代枠)は UD-Q4_K_XL**。品質最優先スロットのため unsloth Dynamic の推奨バランス量子化を使う
- **配布元は「モデル提供元公式 > ggml-org > unsloth/bartowski」の優先順**で、野良量子化は使わない

**検討した代替案と却下理由**:

- **Qwen3.5-4B を bartowski 版(Q4_K_M 2.81GiB)**: 信頼性は同等だが unsloth 版(2.55GiB)の方が tenets の想定サイズに合い、他モデルと配布元が揃う
- **gemma-4-E4B 非 QAT 版**: QAT 版より 0.7GiB 大きく品質根拠も弱い
- **Q5/Q6/Q8 系**: 常駐 4 本合計が VRAM 収支(~26GB / iGPU 32GB、tenets §3.3)を超える
- **Ollama 配布形式**: llama.cpp server 一本化(tenets §3.4)に反する。旧 Ollama はこのマシンからも撤去済み

## Consequences

### Pros

- eval/ の A/B が「配布元・量子化まで固定した再現可能な比較」になる(model_meta に repo:quant を記録すれば [[ADR00002]] のイベント追記と合わせて系譜が完全に残る)
- 全モデルが pin 済み llama.cpp(b9859)の Vulkan で動作確認済み。CPU フォールバック無し
- ディスク実測 65GB / VRAM 常駐見込み ~22.5GB + KV で、収支計画内

### Cons / Tradeoffs

- 提案層(Qwen3.5-4B 25.1 t/s / gemma E4B 27.6 t/s)は LFM2.5 より大幅に遅い。MoA の 3 系統提案を全部走らせると提案フェーズの所要時間はこの遅い側が支配する — 編成の実効レイテンシは M1 で要実測
- 交代枠 2 本(41.5GB)は eval 決着まで併存。敗者を消すまでディスクを食う([[ADR00004]])
- 量子化の品質影響は未計測(速度のみ実測)。品質は eval/ の宿題

## Revisit Triggers

- eval/ の品質 A/B で量子化起因の劣化が疑われたら、該当モデルのみ Q5/Q6 へ上げて VRAM 収支を再計算
- 提案層の遅さが MoA の実用性を損なうなら、提案層の 3 系統目を LFM2.5 兼務(tenets §3.3 の選択肢)に寄せる
- llama.cpp pin 更新時は本表の全モデルで `llama bench` を再実行し、速度退行(4 倍差の前例あり — tenets §9)を検知してから更新する
- モデル追加・交代は「Web 調査で配布元/量子化を選定 → 事前 DL → Vulkan bench → eval → tenets 反映」の本 ADR と同じ手順で行う
