# Plecto を公式イメージ公開までコミット pin のソースビルドで運用する

## Context

tenets §3.5 の Phase 1(素通しリバースプロキシ)を開始するにあたり、Plecto には公式のコンテナイメージ配布が無い(2026-07-02 時点、リポジトリに Dockerfile も無い)。また実運用で以下の制約を踏んだ:

- 依存に `aws-lc-sys` があり、ビルドに cmake + C toolchain が必要。コールドビルドは開発機の Docker 内で約 8 分
- `plecto` バイナリの listen アドレスは positional 引数のみで、デフォルト `127.0.0.1:8080`。コンテナでは `0.0.0.0` バインドが必須
- TLS 証明書パスは manifest 相対。manifest は設定として ro マウントしたいが、開発用証明書は実行時に生成したい
- `Alt-Svc` が広告する HTTP/3 ポートはリッスンポート由来のため、コンテナ内部ポートと公開ポートがズレると h3 アップグレードが黙って壊れる

これらの回避策の置き場所と更新運用を固定する必要があった。詳細な観測記録は draft_0003(Plecto フィールドレポート、gitignore 対象)にあり、改善提案は Plecto 側へ issue 起票する。

## Decision

**moka-1 側の `plecto/Dockerfile` でコミット pin(`PLECTO_REF`)のマルチステージソースビルドを行い、運用上の制約は entrypoint と compose の 4 点セットで吸収する。**

1. **ビルド**: `rust:1.96.1-slim-trixie` で `cargo build --release -p plecto-server` → `debian:trixie-slim` に `plecto` バイナリのみコピー(最終イメージ 189MB)。pin の更新は意図的に行い、踏んだ問題は Plecto 側に issue/ADR 起票する(ドッグフーディングの本義)
2. **listen**: entrypoint が `0.0.0.0:443` を引数で渡す(`PLECTO_LISTEN_ADDR` で上書き可)
3. **証明書**: entrypoint が起動時に自己署名(EC P-256、SAN: moka.localhost / localhost / 127.0.0.1)を named volume に生成。manifest 相対パス仕様に合わせ、`./plecto` を `/run/plecto` に ro マウントし `certs/` だけ named volume を重ねる(mountpoint 用に空の `plecto/certs/` をリポジトリに置く)
4. **ポート**: ホスト側も 443:443(tcp/udp)で揃え、`Alt-Svc: h3=":443"` と公開ポートを一致させる

**検討した代替案と却下理由**:

- **Plecto 側の公式イメージ配布を待つ**: Phase 1 の開始がブロックされる。ドッグフーディングの目的(未成熟部分の露見)にも反する
- **ビルド済みバイナリをリポジトリ同梱**: サプライチェーン的に最悪(検証不能なバイナリを git に置く)。Plecto 自身が cosign 署名文化なのに矛盾する
- **ホストで cargo install して直接常駐**: 「1 コマンド、少数コンテナ」(tenets §1.1)に反する。コンテナ外の常駐物を作らない
- **内部 8443 で listen して 443:8443 マッピング**: Alt-Svc の h3 広告が `:8443` になり、公開側 443 と不一致で HTTP/3 が壊れる。広告ポートの上書き設定が Plecto に入るまでは 443:443 で揃えるしかない

## Consequences

### Pros

- Phase 1 を当日中に開始できた(E2E 検証済み: HTTP/1.1・HTTP/2 ネゴシエーション、path ルーティング、h3 広告、admin エンドポイント)
- コミット pin により「どの Plecto で何が起きたか」がフィールドレポートとして正確に残る
- 証明書自動生成で One-kick(tenets §2-1)を維持。本物の証明書への差し替えは同パスに置くだけ

### Cons / Tradeoffs

- pin 更新のたびに約 8 分のリビルド(BuildKit キャッシュで 2 回目以降は短縮)
- entrypoint / ポート揃え / certs volume の 3 つの回避策は、Plecto 側に対応が入れば不要になる暫定物。回避策であることをコード内コメントで明示済み
- ビルド依存(rust イメージ、crates.io)がビルド時にネットワークを要する

## Revisit Triggers

- Plecto が公式イメージ(または Dockerfile / ビルド済み配布)を公開したら移行する(draft_0003 §3.1 で起票予定の提案)
- listen アドレスの env var / manifest 化が入ったら entrypoint を縮退
- Alt-Svc 広告ポートの上書き設定が入ったら 443:443 制約を解除(rootless Docker 対応が可能になる)
- Phase 2(WASM フィルタ)で `plecto/filters/` のビルドが加わったら、Dockerfile の構成を再評価
