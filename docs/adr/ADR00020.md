# llm サービスの複数モデル提供を llama-server router mode に一本化する

## Context

tenets §3.3 は5モデル常駐編成(高速パス / 提案層×2 / 集約層 / Embedding)を確定しており、[[ADR00007]] は `-c 8192` 制限下で5モデル同時常駐が成立すること(VRAM 45.9/48.0GiB・性能劣化ほぼゼロ)を実測済みだった。しかしあの実証は eval/ がモデルごとに ad-hoc コンテナ(`docker compose run`、ポート別)を立てた測定であり、**本番 compose の `llm` サービスは1プロセス1モデル(Qwen3.5-4B のみ)のまま**で、複数モデルをどう提供するかはどの ADR にも書かれていなかった。

M2(RAG)は Embedding モデル([[ADR00008]])と集約層 gpt-oss-20B の常駐を必要とするため、この空白を埋める決定が前提になる。制約は2つ:

- **Tenet 3(常駐サービス上限 = 5)**: llm 系コンテナの増設は原則衝突する
- **One-kick / OSS**: クローンした第三者の環境でも `docker compose up -d` 一発で再現できること

決定の前提となる事実として、llama.cpp の llama-server に **router mode** が入った: preset ファイルにモデルを宣言し、モデルごとに子プロセスへ隔離、`--models-max` で同時常駐数を制御、OpenAI 互換 API の `model` フィールドでルーティングされる。上限到達時は LRU でモデルを退避する。

## Decision

**`llm` コンテナ1個のまま、llama-server の router mode + リポジトリ内 preset ファイル(ro マウント)で全役割を提供する。**

- M2 時点の常駐セットは3枚: 高速パス Qwen3.5-4B + Embedding Qwen3-Embedding-0.6B + 集約層 gpt-oss-20B(~15GB)
- モデル定義・フラグ(`--embedding --pooling last`、`-c` 等)は preset ファイルに集約し、[[ADR00006]] の配布元・量子化 pin をそこで維持する
- VRAM 収支は開発機(HX370 / iGPU 48GiB carve-out+GTT)を**参照構成**とした値であり、より小さい GPU では router mode の LRU 退避で縮退して動く。OSS 利用者への配慮であり、フェイルソフト(tenets §2-6)と同じ方向の設計
- tenets §3.3 に 2026-07-19 付でサービング形態の段落を追記済み(本 ADR が詳細)

検討した代替案と却下理由:

- **entrypoint 差し替えで複数 llama-server プロセスを1コンテナに同居**: 却下。公式イメージの素の使い方から外れ、プロセス監視・healthcheck・ポート管理が自前になる。イメージ pin 原則(更新は eval/ 実測とセット)の下で自前レイヤーを増やすのは負債
- **llm-chat / llm-embed へのコンテナ分割 + Tenet 3 の数え方改定**: 却下。「llm 系は合算1枠」という例外を作った瞬間に常駐上限の原則が侵食され始める。Tenet はカウント方法の恣意的変更で守るものではない
- **M2 では embedding 専用の2プロセス目だけを暫定追加**: 却下。M3(提案層・集約層の常駐)で同じ問題に戻るだけの先送りであり、暫定形態の撤去コストを追加で払うことになる

## Consequences

### Pros

- Tenet 3(常駐上限5)と One-kick を崩さずに複数モデル常駐へ移行できる
- モデル編成の単一ソースが preset ファイルになり、pin([[ADR00006]])・役割対応(tenets §3.3)がリポジトリ内で完結する
- 小メモリ環境では LRU 退避により自然縮退し、参照構成では全常駐でフル性能 — 同一構成ファイルで両方に対応できる
- 交代枠(Qwen3.6-35B-A3B)のロード/アンロードも router mode のモデル管理に乗せられる見込み(M3 で検証)

### Cons / Tradeoffs

- pin 中の b9859 で router mode と embedding モデルの同居が使えるかは**スパイク未了**。不可なら pin 更新が必要で、規約どおり eval/ の速度実測(バージョン差で数倍の事例あり)を伴う
- 子プロセス隔離のぶんメモリのベースコストがモデル数分乗る(共有はされない)
- healthcheck が router 親プロセスの health に一本化され、個別モデルの死活は API 経由でしか見えない
- LRU 退避は暗黙のレイテンシ源になりうる(退避されたモデルへの初回リクエストがロード時間を払う)。参照構成では全常駐で回避するが、小メモリ環境の体感は未計測

## Revisit Triggers

- スパイクの結果、b9859 で router mode / embedding 同居が不成立だった(→ pin 更新 + eval/ 実測、または本決定の再検討)
- M3 で提案層を含む5枚常駐 + 交代枠スワップを router mode に載せた際、LRU 制御が交代枠の意図(週次バッチ時のみロード)と噛み合わなかった
- llama.cpp 側で router mode の仕様が大きく変わった、または非推奨になった
- 実運用で子プロセス隔離のメモリオーバーヘッドが VRAM 収支(tenets §3.3)を脅かした
