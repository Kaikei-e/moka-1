# Cloudflare Tunnel を任意オーバーレイとして導入する

## Context

moka-1 は個人利用規模のセルフホストが前提だが、外出先など Plecto の 443 番ポートを直接インターネットに公開せずにアクセスしたい場面がある。Cloudflare Tunnel(`cloudflared`)は Cloudflare 側から見て発信接続のみで到達性を作れるため、ホームネットワークの inbound ポート開放や DDNS 運用を避けられる。

一方でこのリポジトリは OSS 公開されている([[project-moka]] の性質)。個人の Cloudflare アカウント資産(トンネルトークン、将来的な公開ホスト名)をリポジトリにコミットするわけにはいかない。また `docs/tenets/moka-tenets.md` §1.2 は常駐サービス数の上限を 5(Plecto / moka-core / moka-web / db / llm、いずれもアプリ層)と定めている。`cloudflared` は取得・濃縮・提示のいずれのドメインロジックも持たない外部接続経路(インフラ層)であり、上限が数える対象の性質と異なる。この整理を明文化せずに `compose.yaml` へ追加すると、上限の意図(アプリ層の複雑さの歯止め)と手段(常駐コンテナ数のカウント)が混同されたまま運用されるおそれがある。

## Decision

**`cloudflared` を `compose.yaml` 本体に含めず、`compose.tunnel.yaml` という任意のオーバーレイファイルに分離する。** 使う場合のみ `docker compose -f compose.yaml -f compose.e2e.yaml ... ` と同じパターンで `-f compose.tunnel.yaml` を足して起動する。使わない環境では `cloudflared` は存在すらしない。

1. **上限との関係**: `cloudflared` はインフラ層(外部接続経路)として扱い、tenets §1.2 の常駐サービス上限 5(アプリ層の内訳)の勘定に含めない。上限が歯止めをかけたいのは「取得→濃縮→提示」ループのドメインロジックが増殖することであり、接続経路の選択(Tunnel か直接公開か)はそれとは別軸の判断
2. **管理方式**: Cloudflare Zero Trust ダッシュボードでトンネルを作成する「トークン方式」を採用。Public Hostname のルーティングと TLS(No TLS Verify)設定はダッシュボード側に置き、リポジトリには接続先ホスト名を書かない
3. **トークンの受け渡し**: Docker secrets(`secrets/cloudflare_tunnel_token.txt`)経由。`cloudflared` は `TUNNEL_TOKEN_FILE` 環境変数でファイルパスを受け取る(2025.4.0 以降の機能。CLI 引数や生の環境変数に直書きすると `docker inspect` / `ps` で露見するため避ける)。既存の `secrets/*` gitignore パターンでコミット対象外になる
4. **イメージ pin**: `cloudflare/cloudflared:2026.6.1@sha256:6d91c121b803126f7a5344005d17a9324788fc09d305b6e2560ec6040a7ae283`。タグ + digest 併記は [[ADR00016]] の Plecto pin 運用に合わせた

**検討した代替案と却下理由**:

- **`compose.yaml` に直接常駐サービスとして追加する**: tenets §1.2 の上限カウントと衝突する。上限を「インフラ層は対象外」という例外付きでその都度解釈させるより、compose ファイルレベルで分離した方が読み手に自明
- **credentials.json(ローカル管理トンネル)方式**: ingress ルールをリポジトリの YAML で管理できる柔軟性はあるが、`cert.pem` / `credentials.json` / `config.yml` の 3 点が増え、gitignore 対象・秘密情報の種類が増える。個人利用規模で複数 Public Hostname を細かく制御する要求が今のところ無いため、トークン方式のシンプルさを優先
- **ホスト側で `cloudflared` を直接起動する(Docker 外)**: repo の compose 構成に一切触れずに済むが、`docker compose up -d` 一発起動(tenets の One-kick 原則)から外れ、`up`/`down` のライフサイクルが分断される。オーバーレイなら one-kick の作法を保ったまま任意参加できる
- **plecto コンテナに `cloudflared` を同居させる**: 公式 Plecto イメージは distroless([[ADR00016]])でシェル・プロセスマネージャが無く技術的に不可能

## Consequences

### Pros

- `compose.yaml` 本体・tenets の常駐上限カウントに一切影響しない。使わない環境(大多数の OSS 利用者)には何の変更も見えない
- トンネルトークンのみが秘密情報で、既存の secrets/ 運用パターン(ファイルベース、gitignore 済み)にそのまま乗る
- `compose.e2e.yaml` と同じオーバーレイパターンなので、リポジトリ内で読み方が一貫している

### Cons / Tradeoffs

- Public Hostname のルーティング設定がダッシュボード側(リポジトリ外)にあるため、「今どこに何が公開されているか」がコードから読めない。個人運用が前提のうちは許容するが、共同運用が必要になれば見直しが要る
- トークン方式は Cloudflare 側の UI 操作に依存し、Infrastructure as Code として完結しない(Terraform 等での宣言的管理は今回のスコープ外)

## Revisit Triggers

- 複数の Public Hostname / ルーティングルールを持つようになり、ダッシュボード操作だけでは追跡できなくなったら、credentials.json 方式(config.yml でルールをリポジトリ管理)へ切り替える
- Plecto に distroless のまま軽量プロセスの同居が可能な仕組み(sidecar container 的な仕掛け)が入ったら、同居案を再評価する
- moka-1 が個人運用を超えてチーム運用になったら、トンネル設定を Infrastructure as Code(Terraform Cloudflare provider 等)に寄せるか検討する
