# 開発機のGPUメモリをcarve-out最小+GTT大盛り構成へ移行する

## Context

旧構成は BIOS UMA carve-out 48GiB(専有VRAM)+ GTT 7.6GiB で、ホスト側 RAM が実質 16GB しか残らなかった。ADR00007 の計測過程でこの構成の実害を2つ実測している:

1. **ホストRAM起因のOOM**: KVデフォルト確保での5モデル同時ロード中にカーネル OOM killer が発動(2026-07-03)
2. **ページキャッシュ不成立**: 21GB級GGUFがキャッシュに乗らず、スワップ枠のロードが毎回ディスク読み直しで 38〜42 秒

一方、調査(`eval/results/research/uma-vram.json`)で以下が判明していた:

- Mesa/RADV は APU では「carve-out と GTT はほぼ同性能」とソースコメントで明言しており、DEVICE_LOCAL 確保に常に GTT フォールバックを付ける(=スピルは自動かつ同速)
- Strix Point/Halo 界隈のローカルLLM標準構成は「carve-out 最小 + `amdgpu.gttsize`/`ttm.*` で GTT 大盛り」
- GTT ヒープは 64GiB 未満に留める(Vulkan ヒープ分割の不具合報告)

## Decision

**BIOS UMA carve-out を 4GiB に縮小し、カーネルパラメタで GTT を 48GiB に拡大する。**

```
amdgpu.gttsize=49152 ttm.pages_limit=12582912 ttm.page_pool_size=12582912
```

(gttsize は MiB、ttm.* は 4KiB ページ単位でいずれも 48GiB。`GRUB_CMDLINE_LINUX_DEFAULT` に追記)

移行前後を同一手順(`moka-eval probe --ctx 8192` / `loadtime -n 3`)で実測した結果:

| 指標 | 移行前 | 移行後 |
|---|---|---|
| ホストRAM total / available | 15Gi / 10Gi | **58Gi / 55Gi** |
| 5モデル同時常駐(LLM編成+35B) | 45.9GiB(carve-out内) | 46.1GiB(VRAM 4.0 + GTT 42.1) |
| 常駐下 tg: 高速パス / 35B | 67.5 / 23.4 t/s | 66.7 / 22.9 t/s(-1〜2%) |
| **35B ロード時間(2回目以降)** | **38〜42 s(毎回コールド)** | **7.0 s(ページキャッシュ成立)** |
| gpt-oss-20b ロード(ウォーム) | 4-5 s | 4-5 s |

「GTTは同速」の理論どおり推論性能は誤差範囲のまま、ホスト可用RAMが約3.7倍になり、OOMの構造的リスクが消え、スワップ枠の実用ロードが5.7倍速くなった。

検討した代替案と却下理由:

- **現状維持(48GiB carve-out)**: 却下。専有確保に性能上の利点がないことが実測で確認され、OOM・キャッシュ不成立という実害だけが残る
- **GTT を 56GiB 以上に拡大**: 却下。GPU使用可能量は増えるが、64GiB 閾値の不具合報告に近づき、フル使用時のホスト残が薄くなる。現編成の常駐+スワップ合計(~46GiB)に対して 4+48=52GiB で十分
- **`amd_iommu=off` の追加**: 却下(今回は見送り)。一部ガイドが推奨するが、将来 NPU(XDNA)を使う選択肢を閉じる

## Consequences

### Pros

- ホスト可用RAM 10Gi→55Gi。計測・ビルド・DB と LLM が同居してもOOMしない
- 週次リキャップのスワップ運用コストが実質消滅(40s→7s)。「品質最優先バッチはスワップ」(ADR00007)の運用が軽くなった
- GPU使用可能量はほぼ同等(52GiB vs 旧55.6GiB)で、現編成の同時常駐に影響なし

### Cons / Tradeoffs

- GPU使用可能量の理論上限は 3.6GiB 減(55.6→52GiB)。現編成では未使用域
- ホストとGPUがメモリを動的に取り合う構成になったため、ホスト側が大量にRAMを使うワークロード(巨大ビルド等)とLLMフル常駐の同時実行では従来に無かった競合がありうる
- BIOS設定に依存する(マシン移行・BIOS更新時に再設定が必要)。本ADRのパラメタが復元手順を兼ねる

## Revisit Triggers

- 物理RAM増設時(GTT上限の再設計。64GiB閾値の不具合が解消されていれば上限引き上げも)
- Vulkan/amdgpu ドライバ更新で GTT スピル性能特性が変わった時(llama.cpp pin 更新時の `llama bench` 再実測で検知)
- LLM編成の常駐合計が 48GiB に迫った時(GTT拡大 or 編成見直し)
