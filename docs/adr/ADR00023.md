# 問い返しQ&Aの文脈検索クエリにタイトルを連結し、文脈選定のみに相対スコア閾値を導入する

## Context

[[ADR00022]] の Revisit Triggers に「実運用で検索の外れ(取りこぼし・ノイズ)が体感された」という項目があった。本番DB実データ(記事4965件、埋め込み4999件、Q&Aログ3件)でこれが実際に観測された。

`core/internal/rag/answer.go` の `Answerer.searchContext` は、記事Aを開いた状態での質問文をそのままハイブリッド検索(pg_trgm + pgvector、RRF融合 k=60、[[ADR00022]])のクエリとして使い、上位5件(記事A自身を除く)を文脈記事として集約層のQ&Aプロンプトに詰めていた。

実データでの具体例(購読記事の内容が特定されるため個別のタイトル・質問文は伏せる):

- トピック語を含まない一般要約質問(「この記事の要点は？」等)を技術系記事に対して投げたケースが複数件あり、いずれも引用5件が対象記事のトピックと全く無関係だった
- 対照的に、固有名詞を含む具体的質問では、引用5件中大半が同一記事の重複クロール(関連性あり)として機能したケースもあった

原因はクエリ構成にあると判断した。トピック語を含まない質問(要約依頼など)は、質問文単体では pg_trgm・pgvector どちらにも意味的な手がかりが乏しく、検索がノイズを拾う。加えて実装を読むと、文書側embedding(`embed.go` の `EmbedArticle`、`title + "\n" + body` 形式)とクエリ側embedding(質問文のみ)が非対称であり、これも一因だった。

## Decision

**1. 文脈検索クエリに対象記事のタイトルを連結する。** `searchContext` が `Searcher.Search` に渡すクエリを、質問文単体から `article.Title + "\n" + question` に変更した。トピック語のない質問でも記事タイトルの固有名詞・トピック語が検索クエリに乗るようになり、文書側embeddingとの非対称も解消される。`Searcher.Search`(`search.go`)・`fuseRRF`・一般検索エンドポイント(`/api/v1/search`)は無改修。

**2. Q&A文脈選定専用の相対スコア閾値フィルタを追加する。** `dropWeakContexts`(answer.go)が、先頭(最高スコア)ヒットの `contextScoreRatio`(初期値 0.5)未満の RRF 融合スコアを持つヒットを文脈から除外する。フィルタ後0件になっても、既存のフェイルソフト経路(対象記事単独で回答)がそのまま機能する。

検討した代替案と却下理由:

- **HyDE(Hypothetical Document Embeddings)**: Gao et al., "Precise Zero-Shot Dense Retrieval without Relevance Labels", ACL 2023(arXiv:2212.10496)。関連文書がどのようなものか手がかりが無い zero-shot 設定を解く手法だが、moka-1では対象記事Aが既知であり前提がズレる。仮想文書生成によるクエリドリフトのリスクもあり不採用
- **クエリルーティング/意図分類**: Self-RAG(Asai et al., arXiv:2310.11511, ICLR 2024)、Adaptive-RAG(Jeong et al., ACL Anthology 2024.naacl-long.389, NAACL 2024)。汎用オープンドメインQAを想定した手法で、moka-1は「記事を開いた状態での質問」という単一ユースケースに閉じている。ルーティング用の追加LLM呼び出しのコスト(レイテンシ・同一ローカルLLMへのキュー競合)が、打ち手1(タイトル連結)で「弱いクエリ」問題自体が緩和されることを踏まえると正当化しにくく、不採用
- **Anthropic Contextual Retrieval**(anthropic.com/news/contextual-retrieval、2024年9月発表): チャンクへの文脈付与によりインデックス時(オフライン)にチャンク単体の参照曖昧性を解消する手法。moka-1は記事をチャンク分割せず `title + body` を1本のembeddingとして扱っており(`embed.go`)、解く問題自体が現行設計に存在しない。チャンク分割を導入する設計変更時にのみ再検討
- **RRF融合後スコアへの絶対閾値**: RRF原論文(Cormack, Clarke, Büttcher, "Reciprocal Rank Fusion outperforms Condorcet and individual Rank Learning Methods", SIGIR 2009)が、RRFスコアは各検索方式の生スコアを捨てて順位だけを使う設計(関連性の絶対量を表す量として設計されていない)と明言している。Azure AI Search公式ドキュメント(Hybrid Search Scoring (RRF) / Create a Vector Query)も同じ理由でハイブリッド融合スコアへの最小スコア閾値を明確に非推奨としている。この2つの独立ソースの三角測量により絶対閾値は不採用とし、先頭ヒットに対する相対比を採用した
- **足切りの一般検索(`/api/v1/search`)へのグローバル適用**: 一般検索は「pg_trgm側が0件でもpgvector側単独で機能する」ことを意図的な設計としており([[ADR00022]] の相互補完の眼目、`e2e/hurl/core/search.hurl` #6 が契約として検証している)、これをQ&A文脈選定と同じ基準で足切りすると、キーワード重なりの無い自然文クエリで検索UIが0件を返すリスクを負う。一般検索UIはユーザーが目でタイトルを見て判断できるためノイズ耐性が高いが、Q&Aは検索結果を無言でLLMプロンプトに詰めるため精度要求が異なると判断し、変更を `Answerer.searchContext` 内に閉じた

## Consequences

### Pros

- 実装コストが小さい(`answer.go` 内のクエリ組み立てとフィルタ関数のみ、LLM呼び出しの追加なし)。一般検索・`Searcher`/`fuseRRF`・既存E2E(`search.hurl`, `qa.hurl`, `rag_failsoft.hurl`)は無改修・無回帰
- `eval/` の pairwise MoA審判プロトコル(主審 gpt-oss-20b + 副審 Claude、位置スワップ両順一致)による実測(`results/summary/qa-context-relevance.json`)で、"generic" スタイルクエリ6件中、title_concat が5勝・0敗・1分け(baseline勝ちゼロ)。判定は両審判で完全一致(judge_agreement=1.0)、位置一貫性1.0。符号検定 p=0.0625
- フィルタが0件を返しても既存のフェイルソフト経路にそのまま乗り、追加の分岐が不要

### Cons / Tradeoffs

- 実測のサンプル数が小さい("generic" クエリ6件、コーパス64記事)。符号検定 p=0.0625 は慣例的な有意水準(p<0.05)に届いておらず、方向性を強く支持する予備実測ではあるが統計的に確定した証拠ではない。追加の "generic" クエリでの再実測が望ましい
- `dropWeakContexts`(相対スコア閾値フィルタ)自体の効果は、`eval/` 環境が PostgreSQL(pg_trgm/pgvector/RRF)を持たないため直接計測できていない。`eval/context_relevance.py` はコサイン近傍のみで「ハイブリッド検索のベクトル側」を近似しており、タイトル連結(打ち手1)の効果はこの近似で検証済みだが、足切り(打ち手2)は `core/` の Go ユニットテスト・E2Eでのみ検証済み
- `contextScoreRatio = 0.5` は初期値であり、実運用での再実測に基づく調整が前提。RRF融合スコアの分布特性(k=60自体も未実測、[[ADR00022]] 参照)次第で最適値が変わりうる

## Revisit Triggers

- "generic" スタイルクエリを追加実測しても title_concat の優位が崩れる、または符号検定が有意水準に届かない状態が続く場合、クエリ構成の別案(質問+記事タイトルの連結順序、記事の要約冒頭の追加連結など)を再検討する
- `contextScoreRatio = 0.5` が実運用で「文脈が毎回空になる」「明らかに弱い文脈がまだ混ざる」のどちらかに体感的に振れた場合、eval/ の実測で値を調整する
- 記事のチャンク分割(長文記事を複数チャンクに埋め込む設計)を導入する場合、Anthropic Contextual Retrievalを再評価する
- moka-1のユースケースが「記事を開いた状態での質問」の単一形から広がり(複数記事横断のQ&Aなど)、クエリの意図が多様化した場合、クエリルーティング/意図分類を再検討する
