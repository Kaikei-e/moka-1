# 機微情報をファイルベース Docker secrets で各コンテナに渡す

## Context

セットアップ初版は `POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-moka}` と、.env(または開発用デフォルト値)を環境変数で渡していた。環境変数方式は `docker inspect` / `docker compose config` / 子プロセスへの継承でパスワードが露出し、compose.yaml にデフォルト値を書く形は公開リポジトリに秘密の平文を残す。moka-1 は公開リポジトリなので、機微情報の扱いを最初に固定する(ユーザー指示、2026-07-02)。

## Decision

**Compose のファイルベース secrets を使う。**

1. トップレベル `secrets:` で `./secrets/postgres_password.txt` を宣言し、参照サービス(db / migrate / moka-core)に `/run/secrets/postgres_password` としてマウントする
2. コンテナ側は `*_FILE` 規約で受ける: db は `POSTGRES_PASSWORD_FILE`、moka-core の store 層も同じファイルを読んで DSN に合成する。migrate(Atlas)は `_FILE` 非対応のため `sh -c` + `$(cat /run/secrets/...)` で URL に合成する(sh 入りの `-community-alpine` 変種を使用)
3. `secrets/` の実ファイルは gitignore。初回に `openssl rand -base64 24 | tr -d '/+='` で手動生成する(手順は secrets/README.md と README)
4. マシン名等のホスト識別子も機微情報として扱い、コミットされるファイルに書かない(ADR 執筆規約 §2.5 と同じ)

**検討した代替案と却下理由**:

- **.env + 環境変数(初版)**: `docker inspect` で露出、デフォルト値が公開リポジトリに残る
- **開発用デフォルト secret をコミットして One-kick 完全維持**: clone 直後に up 一発が通るが、「secret をコミットする」前例を作る。ユーザー判断で却下し、初回のみ1手順(ファイル生成)増やす方を採った。tenets §2-1(One-kick)の明示的な例外第2号
- **Swarm / 外部シークレットマネージャ(Vault 等)**: 単一ノード個人運用に過剰。サービス数上限5にも反する

## Consequences

### Pros

- パスワードが git 履歴・compose.yaml・環境変数のどこにも平文で残らない
- `/run/secrets/` は tmpfs マウントで、コンテナイメージにも焼き込まれない
- `*_FILE` 規約は postgres 公式イメージの標準で、追加実装が要らない

### Cons / Tradeoffs

- clone 直後の `docker compose up` が secret ファイル生成の1手順を要求する(One-kick の例外)。ファイルが無い場合の compose のエラーは分かりやすいとは言えない
- migrate の URL 合成のため Atlas イメージが distroless でなく alpine 変種になる
- パスワードは DSN に埋まるため URL-safe な文字に限る運用制約(生成コマンドで担保)

## Revisit Triggers

- 機微情報が増えて手動生成の手順が3つを超えたら(bootstrap スクリプト or 初回生成の one-shot ジョブ化を評価)
- moka-1 を複数ノード運用に広げるなら(Swarm secrets / 外部マネージャを再評価)
- compose が secret の自動生成(`x-generate` 等)を正式サポートしたら
