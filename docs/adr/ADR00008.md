# Embeddingモデルを Qwen3-Embedding-0.6B(Q8_0)に選定し pgvector を1024次元で運用する

## Context

Embeddingモデルは完全未選定で、`db/schema.sql` の `article_embeddings.embedding` は次元未固定・HNSW未作成のまま先送りされていた(tenets §8-1)。RAG Q&A と関連記事検索(日本語主体)の基盤になるため、モデルと次元を決める必要がある。

候補調査(`eval/results/research/embedding-candidates.json`、出典URL付き)の結論:

- 日本語Retrievalの実測最強は ruri-v3 系(JMTEB Retrieval 81.89)だが、**ADR00006 の配布元縛り(公式 > ggml-org > unsloth/bartowski)を満たす GGUF が存在しない**
- 縛り内で llama.cpp b9859 で確実に動くのは: Qwen3-Embedding-0.6B / 4B(Qwen公式GGUF)、bge-m3(ggml-org)
- Qwen3-Embedding-0.6B の JMTEB Retrieval は 72.81(第三者実測)で縛り内最良。既知の llama.cpp issue #14234(出力破損)は EOS 未付与が原因で、公式GGUFは `add_eos_token=true` / `pooling_type=last` をメタデータに焼き込み済み(ヘッダ実測確認)のため b9859 では追加対策不要
- EmbeddingGemma / multilingual-e5 は日本語性能を理由に候補から除外

自前実測(`eval/`、凍結30記事コーパス+手書き60クエリ(キーワード型/自然文×gold付き)、`eval/results/summary/embed-*.json`):

| モデル | recall@1 | recall@3 | MRR | 備考 |
|---|---|---|---|---|
| Qwen3-Embedding-0.6B Q8_0 | 0.983 | 1.0 | 0.992 | 1024/768/512次元とも同値(MRL切り詰め無劣化) |
| Qwen3-Embedding-4B Q8_0 | 0.983(2560d)/ 1.0(1024d) | 1.0 | 0.992-1.0 | |
| bge-m3 Q8_0 | 1.0 | 1.0 | 1.0 | BERT系のため `-ub 8192` 必須(既定512では500エラー) |

**3候補とも天井に張り付き、この規模のローカル評価では識別不能**(RSS記事は相互に特徴的で検索が易しい)。よって選定はローカルスコアではなく、外部ベンチ+運用要因で行うのが誠実な判断。

## Decision

**`Qwen/Qwen3-Embedding-0.6B-GGUF:Q8_0` を採用し、pgvector 列は `vector(1024)`(ネイティブ次元)で運用する。**

- 起動: `--embedding --pooling last -ngl 99 -c 8192 -ub 8192`
- クエリ側のみ instruction prefix(`Instruct: Given a web search query, retrieve relevant passages that answer the query\nQuery: `)、文書側は素のまま
- 採用理由: 縛り内で日本語Retrieval最良(JMTEB 72.81)+ モデル公式GGUF(配布元優先度1位)+ **MRL対応**(将来4Bへ格上げしても 2560→1024 切り詰めで列互換を維持できる。512次元まで無劣化は自前実測済み)+ 0.64GB で常駐コスト無視可能

検討した代替案と却下理由:

- **bge-m3**(ggml-org、MIT): 次点として記録。prefix不要で運用最簡・ローカル満点だが、MRL非対応で将来のモデル差し替え柔軟性に欠け、日本語外部ベンチでも Qwen3-0.6B の下。乗り換え保険として採用手順を eval/ に残す
- **Qwen3-Embedding-4B**: 却下(現時点)。多言語ベンチでは 0.6B を上回るが日本語での第三者実測がなく、ローカルでも差が出ない。バックフィルのスループット劣化だけ確実に払うことになる。MRL経由で列互換のため、必要になったら再評価
- **ruri-v3-310m 自前GGUF変換**: 保留(watch最有力)。JMTEB 81.89 と大差の性能だが公式GGUFがなく、b9859 の modern-bert 対応で自前変換が通るかは未検証。変換スパイクが成功し sentence-transformers 出力との cos 類似 ~0.99 を確認できたら全候補を置き換える価値がある
- **1024未満への次元削減**: 却下。512まで無劣化だったがコーパス30記事の天井上での測定であり、削減益(ストレージ/速度)も現規模では無意味。1024 は bge-m3 とも互換で保険が効く

`db/schema.sql:100` の `vector(1024)` 固定と HNSW 作成は Atlas 移行を伴うフォローアップ作業とし、本ADRは決定の記録まで(モデル差し替え時は全再埋め込みが必要な点は変わらない)。

## Consequences

### Pros

- 公式GGUF・0.64GB・llama.cpp b9859 で追加対策なしに動く(EOS/pooling焼き込み実測確認)
- vector(1024) は bge-m3(次点)とも 4B(格上げ)とも列互換 — スキーマ変更なしにモデル交換の選択肢を保てる
- retrieval評価ハーネス(recall@k/MRR/MRL再スコア)が eval/ に残り、候補追加時に数分で再実測できる

### Cons / Tradeoffs

- ローカル評価が天井で判別力を持たなかった(コーパス30記事・60クエリ)。実運用の蓄積記事(数千件規模)でこそ差が出る可能性があり、その時の再評価はフォローアップ
- 日本語最強クラス(ruri-v3系)は配布元縛りで見送り — 縛りが性能上限を作っている自覚を持つ
- クエリ側 prefix の付け忘れがバグ源になりうる(moka-core 実装時に `internal/llm` へ集約すること)

## Revisit Triggers

- ruri-v3 の公式GGUF登場、または upstream `convert_hf_to_gguf.py` への ModernBertModel 対応マージ(2026-07-03 のスパイクで b9859/master とも変換クラス不在を確認 — `eval/results/summary/ruri-v3-conversion-spike.json`)
- 実運用コーパス(数百〜数千記事)で recall 劣化・関連記事の外れが体感された(その時は実データで再実測)
- Qwen3-Embedding 後継や JMTEB 上位の公式GGUF付きモデルの登場
- pgvector の次元・HNSW 移行時(フォローアップ実施のタイミング)
