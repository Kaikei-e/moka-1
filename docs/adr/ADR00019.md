# フィード削除をハード削除(CASCADE)+ 二段確認で実装する

## Context

UI/UX 改善の設計 Grill(2026-07-18)で「フィード管理の未成熟」が痛点として確定した。`/feeds` は登録と一覧のみで、購読をやめる手段が UI にも API にも存在しなかった。

削除を導入するにあたり、ADR00002 の不変データモデルとの整合が論点になった。同 ADR はイベント(feed_fetches、article_reads 等)を INSERT-only とするが、**リソース(feeds, articles, tags)の削除までは禁じていない**。実際 `db/schema.sql` は当初から `articles.feed_id` ほか全ての従属テーブルに `ON DELETE CASCADE` を張っており、スキーマ側は「フィードを消せばその店の記録ごと消える」挙動を既に予約していた。

## Decision

**ハード削除を採用する。** `DELETE /api/v1/feeds/{id}` はフィード行を物理削除し、記事・要約・全文・埋め込み・既読の事実は既存の CASCADE で一括して消える。スキーマ変更・マイグレーションは不要。

- **却下した代替案: 購読解除イベント(unsubscribed)で記事を残す案。** イベントを追記して feeds 行を残し、取得スケジューラと一覧クエリから除外する方式。不変データモデルの精神には最も忠実だが、(1) 新テーブルとマイグレーションが必要、(2) 記事一覧・検索・将来のハイライト/RAG まで全クエリに除外条件が波及して複雑化する、(3) そもそも RSS は再購読すれば再取得できる公開データであり、「別れた店の記事が資産として残る」価値が薄い。Grill では「解除した店の記事が残り続けるのはノイズ」という判断で棄却した(ノイズ削減方針)。
- **却下した代替案: 論理削除(deleted_at カラム)。** `updated_at` 禁止と同根の「状態カラムで事実を上書きしない」規約に反するうえ、全クエリへの除外条件波及は購読解除イベント案と同じ。両案の悪いところ取りになるため検討初期で棄却。

UI 側は破壊的操作として扱う。削除は一発では発火せず、**警告様式(DESIGN_LANGUAGE §2.4: geppaku 面 + kindei 1px 枠 + 「注意:」)の二段確認**を挟み、確認文言で「このフィードの記事もすべて消えます」ことを明示する。紺紙金泥ブロックはエラー専用の様式なので確認には使わない。

## Consequences

### Pros

- スキーマ変更ゼロ。CASCADE という既存の設計判断がそのまま活きた(ADR00018 と同じく ADR00002 の貯金)
- 一覧・検索・スケジューラのクエリは一切変更不要。増強系(ハイライト・RAG)にも除外条件が波及しない
- 「消す = 消える」という意味論が読者にとって最も予測可能

### Cons / Tradeoffs

- 誤削除からの復元手段はない(RSS の再登録で記事は再取得できるが、既読の事実・取り寄せた全文・生成済み要約は戻らない)
- `highlights.article_ids` は BIGINT[] で FK が無いため、削除済み記事 ID が配列内に残り得る(ハイライト実装時に解決参照で欠損を無視する設計が必要 — 未解決の負債)

## Revisit Triggers

- 誤削除の実害(消えて困った)が一度でも起きたら、削除前スナップショットか猶予付き削除(ごみ箱)を検討する
- 週次リキャップ・ハイライトが過去記事への参照を持ち始め、CASCADE による欠損が成果物の品質を目に見えて損なうようになったら、購読解除イベント案を再評価する
- マルチユーザー化(現状は単一ユーザー前提)の際は、削除の意味論を所有者単位で再設計する
