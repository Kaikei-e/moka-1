# 記事要約を fulltext 型の自己完結パッケージ・手動キック・防御的 think 除去で実装する

## Context

tenets §3.4 が「moka-1 が足すもの」として挙げる濃縮系機能のうち、要約は最初の実装対象になった。モデル編成は ADR00007 で確定済み(高速パス = Qwen3.5-4B、`chat_template_kwargs: {"enable_thinking": false}`、temperature 0.7 / top_p 0.8 / top_k 20)。DB 側の `article_summaries`(要約成果)・`enrichment_attempts`(試行イベント、kind IN ('summary','tags','embedding'))は ADR00002 のイミュータブルデータモデル初版から存在していたが、実装が伴っておらず未使用だった。

直前の ADR00013(全文取り寄せ)で「fulltext と同型の自己完結ユースケース」というパッケージ規約と、`POST /api/v1/articles/{id}/{feature}` + 冪等(既存 200 / 新規 201)という API 規約がすでに実地で確立していたため、要約もこれに倣うかどうかが最初の論点になった。

実装中に 2 つの実地検証を行った:

1. **llama.cpp の `enable_thinking` フラグの実効性**: pin 済みビルド(`server-vulkan-b9859`, system_fingerprint `b9859-4fc4ec554`)に対し curl で直接 `chat_template_kwargs: {"enable_thinking": false}` を投げたところ、`<think>` タグは一切出力されず、フラグは正しく機能した。一方 Web 調査では、GitHub 上に別ビルド(オートパーサーのリファクタ後)で同フラグが無視される既知 issue(#20182, #20409、2026-07 時点オープン、修正 PR #20329 も未マージ)が見つかった。pin しているビルドはこの回帰の影響を受けないことを実測で確認した。
2. **要約の分量**: 初期プロンプト(「3〜5 文で簡潔に」)で実装・E2E 検証まで完了させたところ、実際に使ったユーザーから「要約文が少し短すぎる」というフィードバックを受け、プロンプトと `max_tokens` を調整した(Decision 参照)。

## Decision

### 1. パッケージ構成: internal/summarize を fulltext 型の自己完結パッケージにする

`core/internal/summarize` を新設し、`Service{store, fullTexts, complete}` というユースケース構造体 + 消費側 interface(`Store`, `FullTextLookup`, `Completer`)+ 具象アダプタ(`HTTPCompleter`)を 1 パッケージに収める、`internal/fulltext` と同じ形にした。LLM チャット補完クライアントも `internal/summarize` 内に直接実装し、汎用 `internal/llm` パッケージへの切り出しは**見送った**。

- **却下した代替案**: tenets §5 のリポジトリ構成案が想定する `internal/{feed,enrich,rag,highlight,llm,store,httpapi}` という汎用バケツ方式(`internal/enrich/summary` のようなサブモジュール)。ADR00013 で実際に生えた「機能名パッケージが自己完結する」規約と食い違うため、tenets の記述より実装の先例を優先した。tenets 側の記述はいずれ実態に合わせて更新が要る(Revisit Triggers 参照)。
- **却下した代替案**: 今のうちに `internal/llm`(チャット補完 + think-strip の共有クライアント)を切り出す案。ADR00008 は embedding クライアントについて「internal/llm へ集約すること」と明言しており将来的には妥当だが、現時点の消費者は summarize 1 つだけであり、2 人目の消費者(タグ抽出・ハイライト等)が出るまで抽出を遅らせる YAGNI 判断をした。think-strip 関数は他のドメイン型に依存しない純粋関数として実装済みで、切り出し自体は将来低コストで行える。

### 2. 起動方式は手動キックのみ

`POST /api/v1/articles/{id}/summary` を読者の明示操作(Web UI の「要約する」ボタンクリック)でのみ起動する。全文取り寄せ(ADR00013)と全く同じ作法にした。

- **却下した代替案(一度実装 → ユーザー指示で撤回)**: フィード登録時に新規記事を非同期 goroutine で自動要約するフックを実装した。tenets §2-2「エージェントが取得→濃縮→提示を無人で回す」という将来ビジョンの前倒し着手のつもりだったが、実装直後にユーザーから「要約は手動キックで」と明示的な訂正が入り、撤去した。撤去に伴い、`InsertArticles` の戻り値を `[]int64`(挿入 ID 一覧)から元の `int`(件数)に戻し、`NewMux`/`handleRegisterFeed` から要約フック関連の引数(`ArticleGetter`, `ArticleSummarizer`, ロガー)も削除した。**教訓**: 「エージェントが自動でやっておく」系の設計は、tenets の思想と実際のユーザー選好が必ずしも一致しないため、実装前に確認を挟むべきだった。常駐エージェントループ(tenets §3.2)を本格導入する際は、この一件を踏まえて自動化の範囲を都度確認する。

### 3. 要約対象テキストの優先順位

`article_fulltexts`(全文取り寄せ済みならそれ)を優先し、無ければ `articles.content`(フィード由来)にフォールバックする(`Service.resolveText`)。要約自体は外部サイトへの新規取り寄せを行わない — 全文取り寄せ(`fulltext.Service.FetchFullText`)とは完全に別の関心事として保つ。

### 4. 長文ガードは明示エラー、黙ってトランケートしない

LLM 呼び出し前にクライアント側で文字数上限(`maxInputChars = 24000`)をチェックし、超過時は `ErrArticleTooLong` を返す。8192 トークンの実効コンテキストからプロンプト・`max_tokens` 分を差し引いた保守的な見積もりによる文字数上限であり、正確なトークナイザは導入していない。

- **却下した代替案**: 超過分を黙ってトランケートしてから要約する案。読者に見えない情報欠落を生むため却下。map-reduce によるチャンキングも v1 では導入しない(RSS 記事は通常この範囲に収まるため、実運用ログを見てから再検討する)。

### 5. think-strip の防御的実装

`<think>...</think>` を機械的に除去する処理(`stripThink`)を実装し、`</think>` が閉じずに truncate された場合は `ErrEmptyCompletion` として扱う(中途半端な CoT を要約として使わない)。`model_meta.think_stripped` フラグで実際に剥がしが発生したかを記録し、ADR00007 の `enable_thinking` フラグが運用上も効いているかを事後追跡できるようにした。

- **却下した代替案**: フラグを信頼して think-strip を実装しない案。Context に書いた通り、pin 済みビルドでは実測上問題ないが、別ビルドでの既知回帰があるため、将来の llama.cpp バージョン更新(ADR00006 の pin 更新方針)に対する安価な保険として残した。

### 6. サンプリングパラメータとプロンプト

ADR00007 確定値(temperature 0.7, top_p 0.8, top_k 20)を使用。システムプロンプトは当初「3〜5 文で簡潔に」だったが、実運用フィードバック(「要約文が少し短すぎる」)を受けて「400 字程度(目安 8〜12 文)、背景・具体的内容・結論を含め、一言二言で終わらせない」に強化し、`max_tokens` も 1024 → 1536 に増量した。実機検証では、記事本文が短ければモデルは正しく短く返す(捏造しない)ことを確認しており、目標文字数は上限的な目安であって強制ではない。

### 7. リトライ・冪等性・失敗記録

リトライ・サーキットブレーカーは導入しない(fulltext 踏襲、ミニマリズム原則)。単発呼び出しで失敗したら `enrichment_attempts`(kind='summary')に成功/失敗を都度記録する。`article_summaries` への保存は DB 側の冪等チェック(`LatestSummary` で既存確認、あれば LLM を呼ばずそのまま返す)による。

- **却下した代替案**: `gobreaker` 等のサーキットブレーカーライブラリの導入。fulltext を含め既存コードに前例が無く、単一ユーザー・ローカル LLM という運用規模では単発呼び出し + 失敗記録で十分と判断した。

## Consequences

### Pros

- fulltext と完全に同じ形なので、次の濃縮機能(タグ抽出・埋め込み)を実装する開発者が迷わない
- think_stripped フラグにより、将来 llama.cpp を更新した際に `enable_thinking` の実効性を実データで再確認できる
- 実機 E2E(Hurl + Playwright、実際の Qwen3.5-4B 呼び出し)で全体フローを確認済み

### Cons / Tradeoffs

- 自動フックを一度実装して撤回したため、`InsertArticles`/`NewMux` 周りに手戻りが発生した(コミット履歴上は吸収済みだが、実装コストとしては残る)
- 汎用 `internal/llm` を作らなかったことで、タグ抽出等の次の濃縮機能を実装する際に summarize パッケージから重複コード(HTTPCompleter 相当)を書くか、その時点で抽出するかの判断が改めて必要になる
- 400 字目標は経験則によるチューニングであり、A/B のような定量評価は経ていない(ADR00007 のような eval/ 実測とは性質が異なる)

## Revisit Triggers

- tenets §5 のリポジトリ構成案(`internal/enrich` 想定)を、実際に生えた `fulltext`/`summarize` 規約に合わせて更新する
- タグ抽出・埋め込み・ハイライト等、2 人目の LLM 消費者を実装するタイミングで `internal/llm` への切り出しを再評価する
- 常駐エージェントループ(tenets §3.2)を本格導入するとき、自動要約フックの要否をユーザーに再度確認してから実装する(このADRの§2で撤回した経緯を踏まえる)
- llama.cpp の pin 更新時(ADR00006)、`enable_thinking` の実効性を再度実機確認し、`think_stripped` の発生率をログから確認する
- 要約の分量・忠実性について実運用ログや読者フィードバックが蓄積したら、eval/ での定量評価(ADR00007 と同じプロトコル)を検討する
- RSS 記事が実効コンテキストを超える事例が実運用で observed されたら、map-reduce 方式のチャンキング導入を検討する
