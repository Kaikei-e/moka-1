# フィード登録 API を /api/v1 バージョニング・冪等登録・SSRF ガード付きで設計する

## Context

M0 の Done 条件は「compose up → フィード登録 → 記事が読める(AI なし)」。moka-core は `/healthz` と `/api/` 501 スタブしか持たず、最初の実 API としてフィード登録と記事一覧が必要だった。参考実装は [Alt](https://github.com/Kaikei-e/Alt) の `alt-backend/app`(rest → usecase → port → gateway → driver の 5 層 + gofeed + SSRF ガード)。ただし moka-1 は「1 コマンド・少数コンテナ・単一言語」制約(tenets §2)が先にあるため、層構造をそのまま移植せず、依存ルールだけを不変条件として運用する(clean-architecture スキル)。

TDD(outside-in)で実装し、E2E(Hurl 5 リクエスト)・ユニット約 40 ケース・CI パリティ(gofmt / go vet / go fix / golangci-lint v2 0 issues / go test)すべて green の状態を本 ADR で固定する。

## Decision

### 1. API は `/api/v1/` でバージョニングする

全エンドポイントに `/api/v1/` プレフィックスを付ける(`POST /api/v1/feeds`, `GET /api/v1/articles`)。Plecto は `/api` を strip せず素通しする契約(plecto/manifest.toml)なので、エッジ側の変更は不要。

- 却下: バージョン無し(`/api/feeds`)。moka-web との同時デプロイなら不要にも見えるが、破壊的変更時の移行パスが無くなる。プレフィックス 1 段のコストは実質ゼロ。

### 2. 登録は冪等(新規 201 / 既存 200)、409 は使わない

`POST /api/v1/feeds` は同じ URL の再登録を エラーにせず、条件付き GET で再取得して新着だけ挿入し 200 を返す。`feeds.url` の UNIQUE + `INSERT ... ON CONFLICT (url) DO NOTHING`(競合時は読み直し)で並行登録も安全。

- 却下: 409 Conflict。単一ユーザーのリーダーで「登録済みだったらエラー」はリトライ・再購読の UX を悪くするだけ。新規/既存の区別はステータスコード(201/200)で伝わる。

### 3. エラーマッピングは 400 / 422 / 502 の 3 分類

| 条件 | ステータス |
|---|---|
| 不正 URL・非 http(s) スキーム・SSRF ブロック | 400(同一バケット — プローブ情報を漏らさない) |
| 取得は成功したがフィードとしてパース不能 | 422 |
| 上流取得失敗(トランスポート・非 2xx) | 502 |

ドメイン層は sentinel(`ErrInvalidURL` / `ErrPrivateHost` / `ErrNotAFeed` / `ErrUpstreamFetch`)を返し、httpapi が `errors.Is` で写像する。

- 却下: SSRF ブロックを 403 で区別する案。攻撃者(将来の公開時)に内部ネットワークのプローブ結果を返すことになる。

### 4. SSRF ガードは既定 ON、`MOKA_FEED_ALLOW_PRIVATE=true` で緩和

`URLValidator` が解決済み IP 全件に対し loopback / private / link-local / unspecified を拒否(IP リテラルは DNS を引かず判定)。リダイレクトは `http.Client.CheckRedirect` で 1 ホップごとに再検証。validate→fetch 間の TOCTOU(DNS rebinding)は既知の残余リスクとしてコードコメントに記録し、dial 時 `Control` 検査は将来課題。

- 却下: スキーム検査のみ(ローカル単一ユーザーだから不要説)。将来の公開・マルチユーザー化で必ず必要になり、後付けは E2E 基盤ごと壊す。env 緩和付きで最初から入れる方が安い。

### 5. 層は機能パッケージに畳む(Alt の 5 層は持ち込まない)

`internal/feed`(ドメイン + ユースケース)/ `internal/store`(pgx アダプタ)/ `internal/httpapi`(ハンドラ)。interface はプロセス外 I/O 境界のみ・消費側定義(`feed.Store`, `feed.Fetcher`, `httpapi.FeedRegistrar`, `httpapi.ArticleLister`)。具象の結線は `cmd/moka/main.go` だけ。`*store.Store` は `ArticleLister` を直接満たし、素通しユースケース層は作らない。パースは gofeed を採用し、その型は `fetch.go` の外に漏らさない。

- 却下: Alt 式の rest/usecase/port/gateway/driver ディレクトリ + gomock。moka-1 の規模では 3 重写経(DTO⇄entity⇄model)と mock 生成基盤が過剰(clean-architecture スキルのガードレール #4)。
- 却下: パーサ自作(encoding/xml)。RSS 0.9x/1.0/2.0 + Atom の方言・文字コード・日付形式を自前で背負う保守コストが gofeed 1 依存より高い。

### 6. 永続化はイミュータブルモデルを踏襲(UPDATE 文ゼロ)

[[ADR00001]] に従い、取得の事実は `feed_fetches` に INSERT-only(成功・304・失敗すべて追記)。条件付き GET の etag / last_modified は最新行から導出。`feeds.title` は INSERT 時に一度だけ設定し、上流のタイトル変更には追従しない(将来必要ならイベントで)。記事は `(feed_id, guid)` の `ON CONFLICT DO NOTHING` で冪等挿入。既存フィードの取得失敗は error イベントを記録してから 502、新規フィードの取得失敗は何も永続化しない(孤児 feed 行を作らない)。

### 7. E2E は compose.e2e.yaml に完全分離

fixture 配信 nginx・`127.0.0.1:8080:8080` 公開・`MOKA_FEED_ALLOW_PRIVATE=true` はすべてオーバーレイ `compose.e2e.yaml` に置き、本番 `compose.yaml` には e2e の痕跡を残さない。実行は `docker compose -f compose.yaml -f compose.e2e.yaml up`(手順は e2e/README.md)。nginx が ETag / Last-Modified を自動付与するため、再登録シナリオが 304 経路を実際に通る。

- 却下: `compose.yaml` 内の `profiles: ["e2e"]` ゲート(初版実装)。起動はされないが本番設定ファイルにテスト用サービスが同居する。「本番に含めない」を優先し別ファイルへ(ユーザー決定)。
- 却下: 実在の外部フィードを叩く E2E。非決定的で CI に耐えない。

### 8. 今は入れないもの

- **グローバルレートリミッタ**(≥5s 間隔、tenets §3.2): ユーザー起点の同期登録は 1 リクエスト 1 取得なので不要。スケジューラループ導入時に `x/time/rate` の単一 Limiter として入れる(Registrar の doc コメントに記録済み)。
- **store の DB 統合ユニットテスト**: testcontainers 基盤が無く、SQL の実挙動は E2E が実 PostgreSQL で担保。ユースケースはポートのフェイクで担保済み。BuildDSN(純ロジック)のみユニットテスト。
- **カーソルページネーション**: M0 は limit(既定 50・上限 200)/ offset で足りる。

## Consequences

### Pros

- M0 のコアパス「登録 → 記事が読める」が E2E で固定された(hurl 5 リクエスト: 201 → 304 経由の冪等 200 → 400 → 一覧 3 件・降順)
- 依存ルールが import で機械的に検査できる(feed パッケージは pgx / gofeed の型を外部に露出しない)
- 追加依存は gofeed と pgx v5 の 2 つだけ。ルーティングは stdlib `http.ServeMux` のまま
- SSRF ガードが env 1 つで本番/開発を切り替えられ、E2E は本番既定(ガード ON)を汚さない
- `.golangci.yml`(v2、bp-go 指定 12 リンタ)が整備され、以後のモジュールも同じ CI パリティで回せる

### Cons / Tradeoffs

- DNS rebinding(validate→fetch 間の TOCTOU)は未対策の残余リスク(公開運用するまでは許容)
- `feeds.title` が登録時点で固定される(上流のタイトル変更に追従しない)
- 登録は同期実行のため、遅い上流だとレスポンスが最大 30s(fetchTimeout)かかる
- ServeMux の仕様上、メソッド不一致の 405 はパスのみパターンの受け皿を手で登録する必要がある(`/api/v1/feeds` 等 2 行のボイラープレート)
- store 層の SQL は E2E でしか実 DB 検証されない(store のロジックが育ったら統合テスト基盤が必要)

## Revisit Triggers

- Plecto 認証フィルタ導入・外部公開を検討する時 → DNS rebinding 対策(dial 時 `net.Dialer.Control` 検査)を必須化し、`MOKA_FEED_ALLOW_PRIVATE` の扱いを再設計
- スケジューラループ(常駐エージェント)実装時 → グローバルレートリミッタ(`x/time/rate`、最小間隔 5s)を導入し、登録時の同期取得もその Limiter を通す
- 記事数が増えて offset ページネーションが遅くなった時(目安: articles 10 万行超) → カーソル方式へ移行
- store のクエリが増えて手書き SQL の維持が辛くなった時 → sqlc 検討 + testcontainers 統合テスト導入
- API の破壊的変更が必要になった時 → `/api/v2` を並走させ、本 ADR の 3 分類エラーマッピングを見直す
- フィードのタイトル変更追従が必要になった時 → UPDATE ではなくイベント(例: feed_title_changes)で対応
