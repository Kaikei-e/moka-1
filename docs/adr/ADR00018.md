# 要約を force フラグで明示的に再生成できるようにする

## Context

ADR00014 で要約機能を実装した際、`POST /api/v1/articles/{id}/summary(/stream)` は `LatestSummary` による冪等短絡を持ち、既存の要約が1件でもあれば LLM を呼ばずそのまま返す設計にした。これは「接続断・タイムアウトからのリトライ」を想定した設計であり、実際に別件で失敗時の再試行ボタン(`RETRY_SUMMARIZE`)も実装済みだった。

ユーザーから別の要求が来た: **要約の生成自体は成功したが、内容の品質に満足できない場合にもう一度生成し直したい**。これは失敗からの回復ではなく、読者が能動的に「やり直す」明示的な操作であり、既存の冪等短絡がそのままでは邪魔になる(何度 POST しても同じキャッシュ済みテキストが返るだけで、実際には何も起きない)。

調査の結果、この要求は ADR00002(不変データモデル)の設計時点で既に見越されていた。`article_summaries` は INSERT-only で、`article_id` にユニーク制約は無く、`store/summaries.go` のコメントにも「再要約 = 追記(最新が有効)」と明記されている。つまりスキーマ側は最初から複数世代の要約を保持できる形になっており、欠けていたのは「読者が明示的にやり直す」ための API・UI 側のトリガーだけだった。

ADR00014 §7 / ADR00015 §3 が「リトライ・サーキットブレーカーは導入しない」「部分テキストは保存しない」と決めたのは、あくまで**失敗時の自動的な再試行**を対象にした話であり、今回の**読者主導の明示的な再生成**とは別の論点であることを確認した上で実装した。

## Decision

### 1. `force bool` パラメータをユースケース層に追加する

`summarize.Service.Summarize`/`SummarizeStream` に `force bool` を追加した。`force` が `true` のときは `LatestSummary` の確認を丸ごとスキップし、既存行の有無に関わらず常に LLM を呼ぶ。生成後の `InsertSummary` は今まで通り追記のみで、既存行を UPDATE・削除することは一切ない。

- **却下した代替案**: 新しいユースケースメソッド(例: `Regenerate`)を別途生やす案。`Summarize`/`SummarizeStream` は「対象テキスト選定 → 長さガード → LLM 補完 → think 除去 → 保存」という一本のパイプラインであり、force は「そのパイプラインの入口で短絡するかどうか」という1点だけが違う。メソッドを分けると同じパイプラインが2箇所に重複するため、既存メソッドへのフラグ追加を選んだ

### 2. API は `?force=true` クエリパラメータで露出する

`POST /api/v1/articles/{id}/summary` と `.../summary/stream` の両方が `?force=true` を受け付ける。付けなければ今まで通り完全に冪等(既存 200 / 新規 201)。付けた場合、既存があっても常に新規生成が起き 201 を返す(新しいステータスコードは導入しない — 実際に新しい行が追記されるので `Created: true` は意味的に正しい)。

- **却下した代替案**: 別エンドポイント(例: `POST .../summary/regenerate`)を新設する案。ADR00015 §1 で「JSON 版と SSE 版は応答形式が違うから分ける」という基準を既に確立しており、force は応答形式ではなく単なる生成方針の切り替えなので、同じ2エンドポイントにクエリパラメータで載せる方が既存規約と整合する

### 3. フロントエンドは「要約をやり直す」という控えめな二次操作にする

`SummaryCard.svelte` は要約成功後、本文の下に `要約をやり直す` ボタンを追加で表示する(DESIGN_LANGUAGE の「記事が主役、AI は給仕」原則に沿い、`.summarize-button` より視覚的に控えめな無枠・無背景のテキストボタンにした。タップターゲットは §8.5 の規律通り 44px を維持)。

再生成が失敗した場合の挙動はユーザー確認の上でシンプルな方を採用した: **直前の(成功していた)要約を消し、既存の失敗表示 + 「再試行する」ボタンにそのまま合流する**。このとき「再試行する」は直前が通常生成だったか再生成だったかを覚えている `lastForce` 状態を引き継いで再送する — でなければ、再生成の失敗を「再試行する」で押し直しても `force=false` で呼ばれてしまい、moka-core 側が保存済みの古い要約をそのまま返すだけで何も変わらない、という混乱を生む

- **却下した代替案**: 再生成中も直前の要約を表示したまま裏でストリームし、失敗時も消さない案。読者から見て「何も悪化しない」利点はあるが、進行中/失敗時の表示用に別バッファ管理が要り実装が複雑化する。DB には旧世代の行がそのまま残るためページ再読み込みで復元でき、実害が小さいと判断し見送った

## Consequences

### Pros

- スキーマ変更・マイグレーション不要(ADR00002 の設計がそのまま活きた)
- 既存の冪等契約(force 無し)は一切変更していないため、既存 e2e(`feeds_and_articles.hurl` 等)は無改修で green のまま
- 失敗時のリトライ導線(ADR済み)と自然に合流し、UI 状態機械が複雑にならなかった

### Cons / Tradeoffs

- `article_summaries` に世代が積み上がるが、読者に「過去の要約一覧」を見せる導線は無い(今回はスコープ外。溜まった旧世代行を掃除する仕組みも未実装)
- 再生成の失敗時に直前の要約が画面から消える(DB には残っているが、読者はページを再読み込みしない限り気づかない)

## Revisit Triggers

- 旧世代の要約行が実運用で無視できない量に積み上がったら、保持世代数の上限や掃除ポリシーを検討する
- 「過去の要約を見比べたい」という要望が来たら、`LatestSummary` を一覧版(`ListSummaries` 相当)に拡張する
- 再生成失敗時に旧要約を消さずに残す設計(却下した代替案)が読者から不満として上がったら、バッファ管理込みで再検討する
