# ブラウザから moka-core への経路を SvelteKit BFF に固定し /api/v1 をブラウザに公開しない

## Context

Web 初期実装(ホーム / 読書ビュー / フィード管理)で、ブラウザと moka-core の間の経路を決める必要があった。前提となる制約:

- 本番では moka-core はホスト非公開で、外部トラフィックはすべて Plecto を通る(tenets、[[ADR00010]] の SSRF ガード前提もこの非公開性に依存)
- moka-web は SSR(adapter-node)で、docker ネットワーク内から `http://moka-core:8080` に直接届く
- moka-web ⇄ moka-core は同一リポジトリ・同時デプロイであり、契約テスト(CDC)を持たない。境界の守りは E2E + Zod スキーマ検証(tdd-workflow)

## Decision

**ブラウザは moka-web としか話さない。** moka-core への読み書きはすべて SvelteKit のサーバー側(`+page.server.ts` / `+layout.server.ts` の load と form actions)が `MOKA_CORE_URL` 経由で行い、レスポンスは `lib/api/schemas.ts` の Zod スキーマで parse してから使う(BFF パターン)。フィード登録はネイティブ form POST(`/feeds` の default action)+ redirect-after-post で、JS 無効でも動く。

検討した代替案:

1. **ブラウザ → Plecto → /api/v1 直**(却下)
   Plecto に moka-core への API ルーティング・CORS 相当の考慮・エラー写像が前提として必要になり、web 実装が Plecto manifest に依存する。E2E も常に Plecto 込みでしか回せなくなる。moka-core のエラー形式(JSON `error` キー)がそのままブラウザ契約になり、UI 文言(moka の声)への写像がクライアントに散る。
2. **ハイブリッド(初期表示は SSR、ミューテーションはブラウザ直)**(却下)
   経路が 2 本になり、SSRF ガード・エラー写像・認可(将来の Plecto 認証フィルタ)を両方で考えることになる。ミニマリズムに反する。

## Consequences

### Pros

- moka-core 非公開の原則が崩れない。Plecto は moka-web への 1 経路だけルーティングすればよい
- moka-core のエラーステータス → UI 文言(§7 の声)の写像が `lib/server/core.ts` の 1 箇所に集まる
- ネイティブ form POST + redirect なので progressive enhancement が既定で成立する
- E2E(Playwright)が moka-web ↔ moka-core の実結合を検証でき、CDC 不在の穴を塞ぐ

### Cons / Tradeoffs

- すべての読み書きが moka-web を経由するホップ 1 回分のオーバーヘッド(ローカル docker ネットワーク内では実害なし)
- 将来のクライアント起点の再取得(要約の再生成ボタン等)も moka-web に API ルート(`+server.ts`)を生やす必要がある
- 登録フォームに `use:enhance` を付けていない(他ルートへの POST を native に寄せた)ため、登録時はフルページ遷移になる

## Revisit Triggers

- SSE によるエージェント進行通知(bp-svelte §12)を入れる時 — `+server.ts` の text/event-stream も BFF 経路の一部として実装できるか確認する
- SvelteKit remote functions が安定化した時(bp-svelte §10)— クライアント起点の再取得を query/command に置き換える評価と同時に、この BFF 境界の形を見直す
- Plecto の認証フィルタ導入時 — 認可の主体が Plecto に移るなら「ブラウザ → Plecto → /api/v1 直」の再評価余地がある
