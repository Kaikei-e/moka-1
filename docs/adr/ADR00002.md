# スキーマをイミュータブルデータモデル(INSERT-only イベント)で設計する

## Context

draft_0002 §4 の当初案は `articles` に `enrichment_status`(pending|summarized|embedded|failed)、`summary`、`tags jsonb`、`embedding` を直接持たせ、濃縮のたびに UPDATE する形だった。スキーマ議論(2026-07-02)で以下が問題になった:

- 単一の `enrichment_status` は線形パイプライン前提で、「要約は成功、埋め込みだけ失敗」という部分失敗が表現できない。要約・タグ・埋め込みの3タスクは本来独立で、fail-soft(tenets §2-6)には部分再処理の表現が要る
- `updated_at` の禁止(「何が更新されたかがわからず、意味のないカラムになる」)を徹底すると、そもそも「行を上書きする」設計自体が情報を破壊していることが見える
- 再要約・ハイライト再生成・埋め込みモデル交代(tenets §8-1)など「やり直し」が予見される機能ばかりで、上書きだと旧結果との比較(eval/ の仕事)ができない

kawasima のイミュータブルデータモデルを設計指針として Web 調査し、次の原則を採った: リソースとイベントを分ける / イベントは日時属性を1つだけ持ち INSERT-only / 状態はカラムでなくイベントの存在から導出する / 訂正は追記(前進修正)。

## Decision

**リソース3テーブル + INSERT-only イベント9テーブルで設計する。**

- **リソース**: `feeds`(購読)、`articles`(取得時点の事実、UNIQUE(feed_id, guid))、`tags`(語彙)
- **イベント**: `feed_fetches`(取得の事実。etag/last_modified は最新行から導出)、`enrichment_attempts`(試行。成功も失敗も追記)、`article_summaries` / `article_tags` / `article_embeddings`(成果。再実行は追記で最新が有効)、`article_reads`(既読。未読=行が無い)、`highlights`(date ごとの最新が有効)、`qa_questions` / `qa_answers`(質問と回答は日時属性1つルールで別イベント)
- **状態の導出**: 「濃縮 pending」= 成果イベントが無い記事、「backoff 中」= 直近の失敗 attempts から計算。`enrichment_status` カラムは持たない
- **禁止事項**: `updated_at` カラム、イベント行の UPDATE/DELETE。時刻が要るなら目的別タイムスタンプ(`fetched_at`, `read_at`, `asked_at`...)

**検討した代替案と却下理由**:

- **単一 `enrichment_status` 列(draft 案)**: 最小だが部分失敗が表現できず、上書きで履歴も消える
- **タスク別ステータス列(summary_status / tags_status / embedding_status)**: 部分失敗は表現できるが、UPDATE 競合・「いつ・なぜ失敗したか」の喪失は残る
- **タグを articles.tags JSONB のまま**: タグ一覧・記事数集計・リネームが SQL で素直にならない。正規化(`tags` + `article_tags`)を採用
- **既読を articles.read_at カラム**: リソース(記事という事実)とユーザー状態の混在。`article_reads` イベントに分離
- **完全なイベントソーシング(CQRS/プロジェクション)**: tenets §3.4 で不採用済み。本 ADR は「テーブル設計を INSERT-only に寄せる」だけで、イベントストア・リプレイ機構は持たない。読みが重くなった箇所だけスナップショット(可変テーブル/マテビュー)を許す現実解の範囲に留める

## Consequences

### Pros

- 部分失敗・部分再処理が行の有無で素直に表現でき、fail-soft と冪等再処理(tenets §3.2)がクエリだけで成立する
- 全成果に系譜(いつ・どのモデルで)が残り、モデル交代時の新旧比較(eval/)がスキーマ変更なしでできる
- UPDATE 競合・lost update が構造的に消える(常駐エージェントと API の同時書き込みで効く)
- `updated_at` 禁止ルールがスキーマレベルで自然に守られる

### Cons / Tradeoffs

- 「最新の要約」等が `DISTINCT ON` / lateral join になりクエリが1段複雑(個人規模のデータ量では性能問題にならない見込み)
- イベントは溜まり続ける。掃除(古い attempts の間引き等)は未設計の負債
- `article_tags` の「タグ外し」は削除イベント未設計(必要になったら `removed` 系イベントを足す前進修正で対応)

## Revisit Triggers

- 「最新を引く」クエリが実測で遅くなったら(まずマテビュー/スナップショット表の追加で対応し、それでも駄目なら該当テーブルのみ可変化を検討)
- イベントテーブルの肥大でディスク・VACUUM が問題になったら(間引きポリシーを設計)
- マルチユーザー化するなら `article_reads` 等のユーザー軸を再設計
