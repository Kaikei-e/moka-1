# Plecto を公式コンテナイメージの digest pin へ移行する

## Context

[[ADR00005]](2026-07-02)は「Plecto に公式イメージが無い」ことを前提に、コミット pin のソースビルド運用を選んだ。同 ADR の Revisit Triggers のうち 3 点が、Plecto のリリースエンジニアリング整備(v0.1.0)で発火・解決した:

1. **公式イメージ配布が始まった**: `ghcr.io/kaikei-e/plecto` へ cosign 署名 + SBOM 付きのマルチアーチイメージが公開されるようになった(タグ `0.1.0` / `0.1` / `latest`)。Plecto 自身のリリースワークフローが、moka-1 の docker build 依存(ADR00005 の Cons)を解消する形で設計されている
2. **listen アドレスが manifest 化された**: `[listen] addr` が manifest の静的な単一ソースになり(Plecto field report §3.2 の反映)、コンテナ起動時に CLI 引数として `0.0.0.0:443` を渡す entrypoint の仕掛けが不要になった
3. **Alt-Svc 広告ポートの上書き**(`[listen] advertised_port`)も追加された。今回のポート構成(443:443)では未使用だが、rootless Docker 対応(内部ポートと公開ポートを分離)が今後可能になった

一方で、公式イメージの最終ステージは完全な distroless(`gcr.io/distroless/cc-debian12:nonroot`、シェル/`openssl` 無し)であり、ADR00005 の entrypoint.sh(起動時に `openssl` で自己署名証明書を生成)はこのイメージの中では動かせない。また `curl` も無いため、compose の `healthcheck:`(`curl -sf http://localhost:9099/readyz`)もそのままでは張れない。`plecto` CLI 自体には `validate` / `schema` / `--version` はあるが、moka-core の `moka healthz` に相当する自己プローブ用サブコマンドは無い。

タグの可変性も実地で観測した: `v0.1.0` タグは一度、指す commit を差し替えられている(`a450010...` → `270d5f69...`、リリースバイナリの実行ビットが `actions/upload-artifact` の転送で失われていたための修正)。pin はタグ単独では不十分で、digest まで固定する必要がある。

## Decision

**`compose.yaml` の `plecto` サービスを `image: ghcr.io/kaikei-e/plecto:0.1.0@sha256:71252ce6061099d2e41681dae14e098183030caa109416d37787b6890e38bca6` に切り替え、証明書生成を新設の one-shot ジョブ `plecto-certs-init` に切り出す。**

1. **イメージ**: タグ(可読性用) + digest(不変性の実体)を併記して pin する。`docker buildx imagetools inspect` で digest を確認してから進める運用(README 明記)
2. **listen**: `manifest.toml` に `[listen] addr = "0.0.0.0:443"` を追加。compose の `command:` は manifest パスのみを渡し、CLI 側の listen 引数は渡さない(manifest が単一ソース)
3. **証明書**: `plecto-certs-init`(`alpine:3.22` ベース、`migrate` と同じ one-shot パターン)が `plecto-certs` named volume に自己署名証明書(EC P-256、SAN: moka.localhost / localhost / 127.0.0.1)を生成する。生成は冪等(既存ファイルがあればスキップ)だが、**所有権の chown は毎回実行**する — distroless の実行ユーザーが nonroot(uid 65532)固定のため、root で生成したファイルをそのままでは読めない(実機検証で `Permission denied` を実際に踏んだ)
4. **healthcheck**: 削除。公式イメージには自己プローブの手段が無いため、Docker Compose レベルの自動ヘルスチェックを諦め、`curl -sk https://localhost/` / `curl http://localhost:9099/readyz` による手動確認に置き換えた(README に明記)。Plecto 側への issue 起票はまだ行っていない(このリリースで解決すべき次の gap として残す)
5. **ポート**: 443:443(tcp/udp)は変更しない。`[listen] advertised_port` によるポート非対称化(rootless Docker 対応)は今回のスコープに含めない

**検討した代替案と却下理由**:

- **pin(コミット SHA)だけ更新してソースビルド運用を維持する**: ADR00005 自身の Revisit Trigger に反する。公式サプライチェーン(cosign 署名 + SBOM 付き証明)の恩恵を受けられず、ビルド時間(pin 更新のたびに約 8 分)も残る
- **タグのみで pin する(digest を省略)**: 今回 `v0.1.0` タグの指す commit が実際に差し替えられるのを観測しており、再現性が保証できない
- **healthcheck を無理に維持する**(例: 別途 curl 入りのサイドカーで代替プローブ): 常駐サービス数を増やす、または compose の複雑さを増す代償が見合わない。distroless を選んだのは Plecto 側の意図的なセキュリティ判断であり、それを崩さずに運用側で吸収する方針を優先した
- **`[listen] advertised_port` を使った rootless 対応まで一緒にやる**: 今回の目的(公式イメージへの移行)から外れるスコープ拡大。ポート構成を変えなくても移行の主目的は達成できるため、別タスクに切り出す

## Consequences

### Pros

- ビルド時間ゼロ(pull のみ)。ソースビルドの ~8 分と rust イメージ / crates.io へのビルド時ネットワーク依存が消えた
- cosign 署名 + SBOM 付きの公式サプライチェーンに乗った(ADR00005 の Cons だった「検証不能なバイナリ問題」を裏返しで解消)
- `[listen]` の manifest 化により entrypoint の CLI 引数渡しが不要になり、compose の記述が単純化した
- 実機検証済み: `docker compose pull/up` でのイメージ取得・起動、`https://localhost/` 200、`/readyz` 200、`/metrics` 取得、moka-core/moka-web 両 upstream の healthy 遷移、SIGHUP ホットリロードをすべて確認

### Cons / Tradeoffs

- **compose レベルの healthcheck を失った**。`docker compose up -d --wait` は plecto の準備完了を待てなくなる(他サービスの healthcheck には影響しない)。手動 curl 確認への後退
- 証明書生成が 2 段(init ジョブ → named volume → 本体コンテナ)になり、entrypoint.sh 単体より部品点数が増えた。ただし `migrate` と同じ形なので運用パターンとしては一貫している
- タグが可変であることが実証された以上、今後の pin 更新でも毎回 digest を再確認する手間が要る(README に手順を明記済み)
- Plecto 側への healthz サブコマンド追加の issue 起票がまだ済んでいない(Revisit Triggers へ持ち越し)

## Revisit Triggers

- Plecto の CLI に `moka healthz` 相当の自己プローブ用サブコマンド(または distroless イメージに軽量プローブバイナリ)が追加されたら、compose の `healthcheck:` を復活させる
- `[listen] advertised_port` を使った rootless Docker 対応(内部ポートと公開ポート 443 の分離)を検討するとき、ADR00005 で却下していた「443:8443 分離」案を再評価する
- Phase 2(WASM フィルタ)で `plecto/filters/` のビルド・配布が要るようになったら、pin 運用(タグ+digest)をフィルタのアーティファクトにも揃えるか検討する
- ghcr.io のイメージ配布が停止・破棄されるようなことがあれば、ソースビルドへ戻る(その場合は ADR00005 の内容がそのまま参照できる)
