# 認証をパスキー1本 + HMAC署名cookie(ステートレス)で実装し、Plecto フィルタは署名検証のみ担う

## Context

認証方式は tenets §8-3 の未決事項だった — 単一ユーザー前提でどこまで簡素化するか。仮説として「パスキー1本 + Plecto フィルタでセッション検証」が置かれていたが、儀式(WebAuthn ceremony)の置き場所とセッション検証の実現方式は未設計だった。

決定を迫る文脈は3つ:

1. **Plecto ドッグフーディング Phase 2**(tenets §3.5): Alt の auth-hub + Kratos スタック相当を「セッション認証フィルタ + レート制限フィルタ」の WASM フィルタ2枚に置換できるかの実験が Phase 2 の主題であり、セッション検証をフィルタで行うこと自体が目的の一部
2. **公開経路の存在**: Cloudflare Tunnel オーバーレイ([[ADR00017]])によりインスタンスがインターネットへ露出しうるため、エッジでの認証は「あれば嬉しい」ではなく必要
3. **OSS 前提**: moka-1 をクローンした第三者が、自分のインスタンスを One-kick(tenets §2-1)で立ち上げ、自分のパスキーを登録できる必要がある。特定ユーザーの資格情報や外部 IdP 契約を前提にできない

## Decision

**パスキー(WebAuthn)1本 + HMAC 署名 cookie のステートレスセッションを採用し、Plecto のセッション認証フィルタは署名検証のみを担う。**

1. **儀式は moka-core**: パスキー登録・ログインは新パッケージ `core/internal/auth`(go-webauthn)が実装し、`/api/v1/auth/*` として BFF 経由([[ADR00011]] — ブラウザに `/api/v1` を直接公開しない)で提供する。資格情報は DB に保存する
2. **スキーマはイミュータブルデータモデル整合**([[ADR00002]]): 資格情報テーブルは INSERT-only。WebAuthn の sign counter は資格情報行を UPDATE せず、assertion イベントの追記で表し最新値を導出する(`updated_at` 禁止規約とも整合)
3. **セッションはステートレスな署名 cookie**: 共有シークレット(新規 Docker secret `session_hmac_key`、[[ADR00003]] のファイルベース方式)で HMAC 署名した有効期限付き cookie を moka-core が発行する。サーバー側セッションストアを持たない
4. **フィルタは検証のみ**: Plecto のセッション認証フィルタ(WASM)は cookie の署名と有効期限を検証するだけ。`/auth/*`(儀式の入口)と `/healthz`(upstream 契約)は検証対象から除外し、無効なリクエストは Fail-closed で遮断する
5. **初回ブートストラップ**: パスキー登録はパスキーが1本も登録されていないときだけ開放する。クローン直後のインスタンスは最初の訪問者(=オーナー)が登録して閉じる

### 検討した代替案と却下理由

- **パスキー + host-kv セッション**(tenets 仮説の字面通り): セッション実体を Plecto の host-kv に置く案。フィルタがログイン応答を観測して kv に書き込む経路が必要になり、それは Plecto の hook 能力(on-response 系)に依存するためスパイクなしでは成立を約束できない。ドッグフーディング密度は最高だが、認証という止まってはいけない機能を未検証の能力の上に置くのは脆い。**host-kv は将来の失効リスト(署名 cookie の弱点補完)用の拡張余地として残す**
- **パスフレーズ1本 + 署名 cookie**: 実装は最小だがフィッシング耐性がなく、tenets 仮説(パスキー)からの後退。公開経路(Tunnel)がある以上、耐性を落とす理由がない
- **Tunnel 層への委譲**(Cloudflare Access 等): エッジ認証を外部サービスに任せる案。フィルタ2枚で auth スタックを置換するという Phase 2 の実験目的そのものが消える。Tunnel はあくまで任意オーバーレイ([[ADR00017]])であり、認証の前提にしない

## Consequences

### Pros

- セッションがステートレスなので、フィルタ⇔アプリ間に共有ストアも書き込み経路も要らない。フィルタは secret ファイル1つで自立して検証できる(Plecto の未検証 hook 能力に依存しない)
- WebAuthn によりフィッシング耐性・資格情報漏洩耐性が単一ユーザー運用でも得られる
- 初回ブートストラップが「最初の登録で閉じる」だけなので、OSS 利用者の One-kick を壊さない(招待トークンや環境変数設定が不要)
- INSERT-only の assertion イベントは監査ログを兼ねる(いつどの資格情報でログインしたかが残る)

### Cons / Tradeoffs

- ステートレス cookie は**即時失効ができない**(有効期限まで生きる)。単一ユーザー運用では許容し、必要になったら host-kv 失効リストで補う(未実装の負債)
- `session_hmac_key` が moka-core と Plecto フィルタの2箇所に配られる共有シークレットになる。ローテーションは両者の再読み込みを要する
- パスキーを紛失すると自力復旧手段がない。復旧は DB の資格情報テーブルを手動で空にして再登録(運用手順としてドキュメント化が必要)
- sign counter を同期しないオーセンティケータ(iCloud Keychain 等は常に 0)ではクローン検知が効かない — WebAuthn 一般の制約

## Revisit Triggers

- Plecto の on-response 系 hook と host-kv 書き込みが安定して使えると実証された(→ host-kv セッション or 失効リストの再評価)
- マルチユーザー化の要求が出た(登録開放ポリシー・セッション設計とも前提が崩れる)
- 署名 cookie の即時失効が実際に必要になる事象(端末紛失等)が起きた
- Phase 4(フィルタの OCI 配布 + 署名)でフィルタへの secret 配布方式が変わる
